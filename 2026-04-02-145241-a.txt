❯ 问题是TianGong的作用两个，核心是修改PineClaw这个如意，项目的代码，最次是负责  
兜底用户输入执行，问题是这个，例如：u用户要给如意增加一个飞书集成的功能，是要写 
代码进入的，或者增加一个自定义工具，你说这个时候怎么办呢？TianGong是负责改项目  
代码的，这个是主要职责，我希望其可以达到自进化的效果                            

⏺ 这个设计很有意思，核心难点一下子清晰了——TianGong 是让 PineClaw 
  能修改自己，也就是自进化。

  我来帮你拆解这个设计的几个关键问题：

  ---
  自进化的核心挑战

  用户说"给如意加个功能"，流程是：

  用户消息
      ↓
  PineClaw Agent（识别：这是需要改代码的需求）
      ↓ tool_call → TianGong
  TianGong 启动 CLI（codex / claude code）
  CLI 修改 src/ 下的代码
      ↓
  ??? 修改完了，功能怎么生效？

  最后那步是整个设计最难的地方，我先把问题列出来再讨论方案：

  难点1：代码改完，新功能怎么上线？

  ┌──────────────────┬──────────────┬──────────────────────────────────────┐
  │       方案       │     效果     │                 代价                 │
  ├──────────────────┼──────────────┼──────────────────────────────────────┤
  │ A. 重启进程      │ 完全生效，稳 │ 服务中断几秒，Docker里可以做到       │
  │                  │ 定           │                                      │
  ├──────────────────┼──────────────┼──────────────────────────────────────┤
  │ B. 热加载（impor │ 无需重启     │ Python热加载有很多坑，只适合改函数实 │
  │ tlib.reload）    │              │ 现，不适合加新文件/类                │
  ├──────────────────┼──────────────┼──────────────────────────────────────┤
  │ C. 只改新增工具  │ 无需重启，适 │ 范围窄，只对Tool有效                 │
  │ ，动态注册       │ 合加Tool     │                                      │
  └──────────────────┴──────────────┴──────────────────────────────────────┘

  我的判断：最合理的方案是分层处理——
  - 新增一个Tool（最常见需求）→ TianGong写好文件 + 动态注册，不重启
  - 改架构级代码（新Channel、新Context模块）→ 改完后通知用户重启，Docker里
  docker restart 一行命令

  ---
  难点2：TianGong的定位——是Tool还是独立模块？

  普通Tool的问题：Tool是被动的，LLM决定调不调用。TianGong承担的是"改造如意自身"
  这件事，应该是更主动的模块。

  我建议 TianGong 同时具备两个身份：
  1. 作为一个 InternalTool 注册进 ToolManager，LLM可以调用它
  2. 但它的实现是一个独立的 TianGongEngine，不是简单的函数

  src/core/tiangong/
      __init__.py
      engine.py      # 管理CLI子进程、收集输出
      evolver.py     # 专门负责"改如意自身"的逻辑（工作目录、动态注册等）
      tool_bridge.py # 作为InternalTool注册进ToolManager的那层胶水

  ---
  难点3：CLI子进程怎么用？

  你说不用SDK，用CLI绑定，这很合理。两个CLI各有特点：

  ┌────────────┬────────────────────────────────┬───────────────────────────┐
  │    CLI     │            适合场景            │      Python启动方式       │
  ├────────────┼────────────────────────────────┼───────────────────────────┤
  │ claude     │                                │ claude --print "task" 或  │
  │ (Claude    │ 修改PineClaw代码，有项目上下文 │ claude --output-format    │
  │ Code)      │                                │ stream-json               │
  ├────────────┼────────────────────────────────┼───────────────────────────┤
  │ codex      │ 通用编码任务                   │ codex "task"              │
  └────────────┴────────────────────────────────┴───────────────────────────┘

  对于修改如意自身这个主职责，Claude Code 更合适，因为它能读懂整个项目结构（有
  CLAUDE.md、目录感知）。

  ---
  我建议的完整流程设计

  用户：给如意加一个天气查询工具

  PineClaw Agent
    └─ 识别意图：需要修改代码
    └─ tool_call("tiangong_evolve", {
           task: "在 src/core/tool/tools/ 下新建 weather/ 工具，
                 实现天气查询，注册进 ToolManager",
           mode: "new_tool"
       })

  TianGong Engine
    └─ 判断 mode=new_tool，走"轻量通道"
    └─ subprocess: claude --print "..." --cwd /app/src
    └─ 等待完成，收集 stdout

  （Claude Code 在 src/ 目录下写好文件）

  TianGong Evolver
    └─ 扫描新文件，发现 weather/definition.py
    └─ 动态 import + 注册进 ToolManager
    └─ 返回："天气查询工具已添加并生效"

  PineClaw Agent
    └─ 回复用户："已为你添加天气查询工具，现在可以直接问我天气了"

  一个需要你决策的问题

  - 还是整个项目目录（包括 CLAUDE.md、config/ 等）

  这个边界决定了"自进化"能做到什么程度，也决定了安全性。你倾向哪个方向？
