# 上下文模块架构总览

## 设计理念

上下文管理是 Agent 的核心。更长的上下文不等于更好的响应——过度加载上下文会导致 Agent 以意识不到的方式失败（污染、干扰、冲突、混淆）。设计重点在于**整理**而非单纯**获取**。

本模块基于 ce-context-management 技能规范重构，核心变化：

1. **ContextItem 富数据结构** — 内部流转使用结构化数据而非原始 dict
2. **分段式系统提示词** — 支持动态注册和管理多个 prompt 段落
3. **双记忆模块** — 长期记忆（飞书文档）与短期记忆（对话历史+压缩）分离
4. **工具上下文管理** — 独立模块管理 tool_calls/tool 响应配对
5. **抽象存储层** — 存储接口化，ChatStore 重构为 JsonlContextStorage

## 核心数据结构：ContextItem

系统内部所有上下文模块使用 `ContextItem` 进行数据流转，它在 OpenAI message dict 的基础上增加了元数据：

| 字段 | 类型 | 说明 |
|------|------|------|
| role | str | "system" / "user" / "assistant" / "tool" |
| content | str \| None | 消息内容 |
| source | str | 来源标识: "system_prompt", "long_term_memory", "history", "tool" 等 |
| priority | int | 1(LOW) ~ 4(CRITICAL)，压缩时高优先级不被移除 |
| created_at | float | 创建时间戳 |
| token_estimate | int | Token 估算值 |
| metadata | dict | 扩展元数据 |
| tool_calls | list | 工具调用列表 |
| tool_call_id | str \| None | 工具响应关联 ID |
| name | str \| None | 工具名称 |

通过 `to_message()` 方法转换为 LLM API 需要的 dict 格式；通过 `from_message()` 类方法从 dict 创建。

## 数据流

```
用户输入
    │
    ▼
SimpleAgent.run()
    │
    ├── ctx.append_message(user_msg)
    │       └── ContextManager 路由 → ShortTermMemoryContext
    │
    ├── ctx.get_context()
    │       └── _build_context_items():
    │           1. SystemPromptContext.format()      → ContextItem[]
    │           2. LongTermMemoryContext.format()     → ContextItem[]
    │           3. ShortTermMemoryContext.format()    → ContextItem[]
    │           4. ToolContext.format()               → ContextItem[]
    │       └── _to_messages():
    │           ContextItem[] → dict[] → sanitize_messages() → LLM
    │
    ├── tool_loop (LLM → tool_calls → execute → response → repeat)
    │       └── intermediate messages → ctx.append_message()
    │           └── ContextManager 路由 → ToolContext
    │
    ├── ctx.archive_tool_context()
    │       └── ToolContext 中的消息归档到 ShortTermMemoryContext
    │
    ├── ctx.append_message(assistant_msg)
    │       └── ContextManager 路由 → ShortTermMemoryContext
    │
    └── ctx.needs_compression() → ctx.compress()
            └── ShortTermMemoryContext.compress()
                └── ContextCompressor.compress_with_llm()
```

## 消息组装顺序

消息序列的顺序直接影响 LLM 的理解质量：

1. **system**（最前）：系统提示词，LLM 视为最高优先级指令
2. **system**（长期记忆）：用户画像和偏好
3. **system**（摘要）：压缩后的历史摘要（如有）
4. **history**：过去的对话记录
5. **tool**：当前轮次的工具调用和响应

## 从旧版本迁移

| 旧模块 | 新模块 | 说明 |
|--------|--------|------|
| ChatStore | JsonlContextStorage | 实现 IContextStorage 接口 |
| ContextManager._messages | ShortTermMemoryContext | 独立的短期记忆模块 |
| ContextManager._summary | ShortTermMemoryContext._summary | 内聚到同一模块 |
| ContextManager.compress() | ShortTermMemoryContext.compress() | 压缩逻辑下沉 |
| MemoryContext | LongTermMemoryContext | 重命名，保留向后兼容别名 |
| BaseContext.get_messages() | BaseContext.format() | 返回 ContextItem[] 而非 dict[] |
