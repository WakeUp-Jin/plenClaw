# 压缩与存储

## 压缩策略

### 概述

上下文压缩的目标：提炼关键信息，释放上下文窗口，在尽量不降低响应质量的前提下继续运行。

**文件**: `src/core/context/utils/compressor.py`

### ContextCompressor

提供两种压缩手段：

#### 1. LLM 摘要压缩 (`compress_with_llm`)

使用 LLM 对旧消息生成摘要，保留语义而非原文。

**流程**:
1. 消息太少（< 4 条）不压缩
2. 从后往前按 token 累加，找到保留比例（默认 30%）的分割点
3. 调整分割点，确保不在 tool_calls/tool 响应中间截断
4. 将要压缩的消息转为文本，用 LLM 生成摘要
5. 返回 `CompressionResult`

**压缩提示词要求**:
- 保留所有关键决策和结论
- 保留涉及的文件路径和代码变更
- 保留未完成的任务和计划
- 保留用户的偏好和约束条件
- 删除重复信息和不重要的对话

#### 2. 工具消息裁剪 (`trim_tool_messages`)

不调用 LLM，直接移除旧的工具调用轮次，只保留最近 N 轮。

**流程**:
1. 识别工具调用轮次（assistant tool_calls + 对应 tool responses）
2. 只保留最近 `keep_last_rounds` 轮（默认 3 轮）
3. 非工具消息不受影响

### 工具调用边界调整

压缩时必须保证 tool_calls 和对应的 tool responses 不被分割到不同区域：

- 如果分割点落在 tool response 上，向前移动到对应 assistant 消息之前
- 如果分割点落在带 tool_calls 的 assistant 上，向后移动到所有 tool responses 之后

### 压缩触发条件

由 `ContextManager.needs_compression()` 检测：

```python
token_usage >= max_token_estimate * compression_threshold
```

默认配置（`CompressionConfig`）:

| 参数 | 默认值 | 说明 |
|------|--------|------|
| max_token_estimate | 60000 | 模型 Token 上限估算 |
| compression_threshold | 0.7 | 使用率 >= 70% 触发压缩 |
| overflow_threshold | 0.95 | 使用率 >= 95% 视为溢出 |
| compress_keep_ratio | 0.3 | 压缩时保留最近 30% |

## Token 估算

**文件**: `src/core/context/utils/token_estimator.py`

不依赖外部 tokenizer 库，基于字符特征快速估算：

| 字符类型 | 估算规则 |
|----------|---------|
| CJK 字符 (U+4E00~U+9FFF) | 1.5 字符 ≈ 1 token |
| ASCII/其他 | 4 字符 ≈ 1 token |
| 每条消息 | +4 token（角色等元数据开销） |

**TokenEstimator** 支持：
- `estimate_text(str)` — 纯文本估算
- `estimate_item(ContextItem)` — 单个 ContextItem 估算
- `estimate_items(list[ContextItem])` — 批量 ContextItem 估算
- `estimate_message(dict)` — 单个 message dict 估算
- `estimate_messages(list[dict])` — 批量 message dict 估算
- `format_tokens(int)` — 格式化显示（如 "12.5K", "1.2M"）

## 消息清理与验证

**文件**: `src/core/context/utils/message_sanitizer.py`

保证发送给 LLM API 的消息符合规范，特别是 tool_calls 和 tool responses 的配对关系。

### sanitize_messages()

自动修复不完整的消息序列：

**规则 1**: 移除不完整的 assistant 消息——有 `tool_calls` 但缺少部分 tool 响应时，同时移除该 assistant 消息和已有的部分响应。

**规则 2**: 移除孤立的 tool 消息——`tool_call_id` 不在任何 assistant 消息的 `tool_calls` 中。

### validate_messages()

只校验不修改，返回 `ValidationResult`:

```python
@dataclass
class ValidationResult:
    valid: bool
    issues: list[ValidationIssue]
```

问题类型：
- `missing_tool_response` — tool_call 缺少对应的 tool 响应
- `orphaned_tool_message` — tool 消息找不到对应的 assistant tool_call

### 触发时机

| 时机 | 方法 | 说明 |
|------|------|------|
| 组装消息时 | `sanitize_messages()` | `_to_messages()` 的最后一步 |
| 工具中断时 | `ToolContext.sanitize()` | 用户取消导致不完整 |
| 调试时 | `validate_messages()` | 只检查不修改 |

## 存储抽象层

### IContextStorage 接口

**文件**: `src/core/context/storage/base.py`

纯 I/O 契约，定义了消息持久化和压缩状态的标准接口：

| 方法 | 说明 |
|------|------|
| `append(message)` | 追加一条消息 |
| `load_all()` | 加载所有消息 |
| `load_from_line(line)` | 从指定行加载 |
| `count_lines()` | 消息总数 |
| `save_checkpoint(summary, line)` | 保存压缩 checkpoint |
| `load_checkpoint()` | 加载 checkpoint |
| `new_session()` | 开启新会话 |

### JsonlContextStorage 实现

**文件**: `src/core/context/storage/jsonl_store.py`

从原 `ChatStore` 重构，实现 `IContextStorage` 接口。

**文件布局**:

```
data/chat_history/
    session_20260311_143022.jsonl            <- 所有消息，append-only
    session_20260311_143022.checkpoint.json  <- 压缩状态（如有）
```

**特点**:
- Append-only JSONL 格式，每行一条 JSON 消息
- 自动查找最新 session 文件或创建新文件
- Checkpoint 与 session 文件同名但后缀不同
- 旧 session 文件永久保留

### 扩展新存储

实现 `IContextStorage` 接口即可接入新的存储后端（如 SQLite、Redis 等）：

```python
class SqliteContextStorage(IContextStorage):
    def __init__(self, db_path: str):
        ...
    
    def append(self, message: dict) -> None:
        ...
    
    # ... 实现其他方法
```
