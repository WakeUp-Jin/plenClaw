# 天工（TianGong）模块 — 需求与架构总览

## 一、定位

天工是如意（PineClaw）的**铁匠**——为如意锻造 CLI 工具。

- 天工**不修改**如意本身的项目代码
- 天工的产物是**可执行的二进制 CLI 工具**
- 如意通过命令行调用这些工具来扩展自己的能力
- 师傅为弟子打造兵器，弟子拿着兵器去干活

```
用户需求 → 如意判断自己缺少某种能力
  → 请求天工锻造工具
    → 天工用 Rust 编写 CLI → 编译为二进制
      → 如意通过 bash 工具调用二进制
        → 如意变得更强
```

---

## 二、核心理念

### 2.1 工具的形态：编译型 CLI 二进制

天工锻造的工具不是 Python 模块，而是 **Rust 编译的单二进制文件**。

为什么选这种形态：

| 维度 | Python tool 文件 | Rust 编译二进制 |
|------|-----------------|----------------|
| 运行时依赖 | 需要如意的 Python 环境 | 零依赖，拿到就能跑 |
| 隔离性 | 和如意共享进程，可能互相影响 | 完全独立进程 |
| 质量保证 | 运行时才能发现错误 | 编译过了几乎没 bug |
| 部署方式 | 需要如意加载 | 放到目录里即可 |
| 使用方式 | 如意内部函数调用 | 如意通过 bash 执行命令行 |

### 2.2 天工本身的定位

天工是一个**极简调度壳**：

- 内部运行 Coding Agent CLI（Claude Code / Codex / Kimi / OpenCode 等）
- 自身不写代码、不编译、不写说明书——这些全由 Coding Agent 按照锻造规范完成
- 天工只做三件事：**巡查锻造令、启动 Coding Agent、监控执行结果**
- 所有"智慧"在 Coding Agent 那边，天工只负责调度

### 2.3 通信方式：锻造令 + 定时巡查

如意与天工之间**不使用 HTTP API**，而是通过文件通信：

- 如意下达"锻造令"（Markdown 文件），写入约定目录
- 天工定时巡查（默认每 15 分钟），发现锻造令就开始执行
- 产物通过共享文件系统交付
- 零网络依赖，纯文件操作，简单解耦

## 三、整体架构

### 3.1 系统全貌

```
用户（飞书 / 微信 / CLI）
  │
  ▼
┌────────────────┐
│  如意（PineClaw）│ ← Agent 核心
│  Channel / Tool │
└───────┬────────┘
        │
        ├─ 自有 Tool（天气、搜索...）
        │
        ├─ bash 工具 → 调用天工锻造的 CLI 二进制
        │              （读 TianGongToolList Skill 知道有哪些工具）
        │
        └─ tiangong_evolve → 请求天工锻造新工具
                │
        ┌───────┴──────────┐
        │   天工容器         │
        │                   │
        │  Coding Agent CLI │ ← Claude Code / Codex / Kimi...
        │  Rust 工作空间     │ ← 写代码、编译、验证
        │                   │
        │  产物 → 二进制文件  │
        │       → 说明书 MD  │
        │       → 更新 SKILL │
        └──────────────────┘
```

### 3.2 调用链路

```
场景 A：如意已有工具
用户："查天气" → 如意有 weather-tool → bash 执行 → 返回结果

场景 B：如意没有工具，一次性任务
用户："压缩这个文件" → 如意没有对应工具 → tiangong_execute → 天工沙箱执行 → 返回结果

场景 C：如意需要新工具
用户："以后能帮我查天气吗？" → tiangong_evolve → 天工锻造 weather-tool → 如意获得新能力
```

天工锻造的工具以 **Agent Skill** 的标准格式交付，如意天然支持读取。

```
.pineclaw/
├── config.json                     ← 如意配置
├── skills/                         ← Skill 目录
│   └── TianGongToolList/
│       ├── SKILL.md                ← 如意读这个，知道有哪些锻造好的工具
│       ├── references/             ← 每个工具的使用说明书（Markdown）
│       │   ├── weather-tool.md

### 4.3 工具说明书示例（references/weather-tool.md）

```markdown
# weather-tool 使用说明书

## 功能
查询指定城市的实时天气信息。

## 用法

​```bash
weather-tool [OPTIONS]
​```

## 参数

| 参数 | 必填 | 说明 | 示例 |
|------|------|------|------|
| --city | 是 | 城市名称（中文或英文） | --city beijing |
| --format | 否 | 输出格式，默认 json | --format json / text |
| --help | 否 | 显示帮助信息 | --help |

## 输出示例

​```json
{
  "city": "北京",
  "temperature": 25,
  "weather": "晴",
  "humidity": 40,
  "wind": "北风 3级",
  "update_time": "2025-01-01 12:00:00"
}
​```

## 错误码

| 退出码 | 含义 |
|--------|------|
| 0 | 成功 |
| 1 | 参数错误 |
| 2 | 网络请求失败 |
| 3 | 城市未找到 |

## 注意事项
- 需要网络连接
- 依赖天气 API，需配置 WEATHER_API_KEY 环境变量
```

### 4.4 工具双重说明机制

每个锻造工具同时提供两种说明方式：

```

└──────────────────────────────────────────────────┘
```

## 五、天工内部架构

### 5.1 天工容器

天工运行在**独立 Docker 容器**中，项目启动时天工容器一同启动。

```
天工容器
├── Coding Agent CLI（可切换）
│   ├── Claude Code
│   ├── Codex (OpenAI)
│   ├── OpenCode
│   └── Kimi CLI
│
├── Rust 工具链（预热模式，常用 crate 已缓存）
│   ├── rustup / cargo
│   └── 预缓存：clap, reqwest, serde, tokio, anyhow...
│
├── 工作空间 /workspace/
│   ├── Cargo.toml
│   ├── src/
│   └── target/release/
│
├── 天工调度器（Python，极简）
│
└── 共享卷挂载
    └── /shared/ → 映射到宿主机 .pineclaw/
        ├── tiangong/orders/     ← 巡查锻造令
        ├── tiangong/forge-spec.md  ← 读取锻造规范
        └── skills/TianGongToolList/ ← 交付产物
```

### 5.2 Coding Agent CLI 接入条件

天工可以接入任何满足以下条件的 Coding Agent CLI：

**必须满足：**

| 条件 | 说明 |
|------|------|
| 非交互模式 | 能通过参数传入任务，不需要人工交互 |
| 指定工作目录 | 能在指定的 Rust 工作空间下工作 |
| stdout 可捕获 | 天工需要拿到执行结果 |
| 退出码规范 | 0=成功，非0=失败 |
| session 恢复 | 支持多轮交互（编译失败后继续修复等场景） |

**加分条件：**

| 条件 | 说明 |
|------|------|
| 读取项目文件 | 能自己读工作空间内的文档、锻造规范 |
| 流式输出 | 天工可以实时报告锻造进度 |
| 权限控制 | 可限制在指定目录内操作 |

### 5.3 天工核心逻辑

天工**不实现任何锻造逻辑**。写代码、编译、写说明书、更新 SKILL.md 全部由 Coding Agent 按照锻造规范自行完成。

天工只做三件事：

```
定时巡查 orders/pending/
  ↓
发现锻造令 → 移到 processing/ → 读取内容
  ↓
拼接：锻造规范 + 锻造令 → 喂给 Coding Agent CLI
  ↓
Coding Agent 自行完成全部锻造工作
  ↓
执行成功 → 锻造令移到 done/
执行失败 → 标记失败原因
```

### 6.3 决策优先级（写入如意的 System Prompt）

```
处理用户请求时，遵循以下优先级：

1. 优先使用自有工具 — 如意已有的工具能完成，直接用
2. 使用天工锻造的工具 — 查看 TianGongToolList Skill，用 bash 调用已有二进制
3. 委托天工兜底执行（tiangong_execute）— 一次性任务，不值得锻造工具
4. 请求天工锻造新工具（tiangong_evolve）— 用户明确需要新的持久能力
```

## 七、部署方案

### 7.1 独立双容器（Docker Compose）

天工始终是独立 Docker 容器，项目启动时通过 Docker Compose 一同拉起。

```
┌─ Docker Compose ──────────────────────────────────────┐
│                                                        │
│  ┌─ 如意容器 ──────────┐    ┌─ 天工容器 ────────────┐ │
│  │ Python               │    │ Rust 工具链（预热）    │ │
│  │ PineClaw Agent       │    │ Coding Agent CLI      │ │
│  │ bash tool            │    │ /workspace/           │ │
│  │                      │    │ 天工调度器（Python）   │ │
│  └───────┬──────────────┘    └──────┬────────────────┘ │
│          │                          │                  │
│          └──── 共享卷 .pineclaw/ ───┘                  │
│                                                        │
│  如意写锻造令 → .pineclaw/tiangong/orders/pending/     │
│  天工巡查执行 → 产物交付到 .pineclaw/skills/            │
└────────────────────────────────────────────────────────┘
```

- 如意容器：纯 Python 环境，轻量
- 天工容器：Rust + Coding Agent CLI 环境，预热常用 crate
- 通过共享卷 `.pineclaw/` 完成所有通信和交付
- 无 HTTP 依赖，纯文件通信

## 八、Rust 选型理由

### 为什么是 Rust 而不是 Go？

| 维度 | Rust | Go |
|------|------|-----|
| 编译产物大小 | 1-3 MB（简单 CLI） | 5-10 MB |
| 运行时依赖 | 零依赖 | 零依赖（打平） |
| 编译速度 | 慢（10-30s） | 快（几秒） |
| AI 写代码质量 | **编译过了几乎没 bug** | 可能有运行时 nil panic |
| 错误处理 | Result<T,E> 强制处理 | 容易忘检查 err |
| CLI 生态 | clap（自动生成 --help） | cobra（同样成熟） |
| 跨平台编译 | 需配置 target | 更简单 |

**核心选择理由：编译即验证。**

天工让 Coding Agent 写代码 → `cargo build` 编译通过 → 交给如意的就是可靠的成品。Rust 编译器的严格是最好的质量检测关卡。

### 天工容器需要预备的 Rust 环境

- Rust 工具链（rustup + stable channel）
- 预缓存常用 crate 的编译结果：`clap`、`serde`、`reqwest`、`tokio` 等
- 这样新工具的编译速度会大幅提升（增量编译）

## 九、安全与边界

| 项目 | evolve 模式（锻造工具） | execute 模式（兜底执行） |
|------|----------------------|----------------------|
| 工作目录 | /workspace/（Rust 工作空间） | /tmp/sandbox/ |
| 产物 | 二进制 + 说明书 → 交付到 Skill 目录 | 无持久产物 |
| 能否修改如意代码 | **不能** | **不能** |
| 后端 CLI | Coding Agent（写 Rust） | Coding Agent（执行命令） |
| 执行时间 | 长（分钟级） | 短（秒级） |
| 风险等级 | 低（不碰如意代码） | 低 |

## 十、未来演进

1. **工具版本管理** — 同一工具可以有多个版本，支持回滚
2. **工具市场** — 标准化的工具 Skill，可以在不同如意实例间共享
3. **自动进化** — 如意通过心跳机制分析自身不足，主动请求天工锻造工具
4. **多语言支持** — 除了 Rust，也可以锻造 Go / Python 打包的工具（根据场景选择）
5. **工具组合** — 复杂任务可以组合多个 CLI 工具，通过管道串联
