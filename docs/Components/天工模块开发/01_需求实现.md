# 天工（TianGong）模块 — 需求实现

## 一、实现总览

天工模块分为两个独立部分：

| 部分 | 位置 | 说明 |
|------|------|------|
| **如意侧** | plenClaw 项目中 | tiangong_evolve 工具（写锻造令文件） |
| **天工容器** | 独立 Docker 容器 | 调度器 + Coding Agent CLI + Rust 工具链 |

两者通过共享卷 `.pineclaw/` 通信，没有任何网络调用。

## 三、锻造令（Orders）

### 3.1 锻造令是什么

锻造令是如意写给天工的 Markdown 文件，描述需要锻造什么工具。天工巡查时读到锻造令，就知道该干什么活。

### 3.2 锻造令文件命名

```
{日期}-{时间}-{工具名}.md
```

示例：`20250405-143000-weather-tool.md`

日期时间前缀保证巡查时按先后顺序处理。

### 3.3 锻造令格式

```markdown
# 锻造令：weather-tool

## 需求描述
查询指定城市的实时天气信息，返回温度、天气状况、湿度、风力等。

## 期望用法

```bash
weather-tool --city beijing --format json
```

## 期望输出

```json
{
  "city": "北京",
  "temperature": 25,
  "weather": "晴"
}
```

## 参数要求
- --city（必填）：城市名称，支持中文和英文
- --format（可选）：输出格式，默认 json，支持 json / text
- --help：显示帮助信息

## 外部依赖
- 天气 API（如 OpenWeatherMap）
- API Key 通过环境变量 WEATHER_API_KEY 传入

## 元信息
- 创建时间：2025-04-05 14:30:00
- 请求方：如意
- 优先级：normal
```

### 3.4 锻造令的生命周期

```
如意写入 → pending/
  ↓
天工巡查发现 → 移到 processing/
  ↓
天工启动 Coding Agent → 锻造中
  ↓
锻造完成？
  ├─ 成功 → 移到 done/，追加完成记录
  └─ 失败 → 移到 done/，追加失败原因

done/ 中的锻造令追加内容示例：

## 锻造结果
- 状态：成功
- 完成时间：2025-04-05 14:52:00
- 耗时：22 分钟
- 二进制位置：scripts/weather-tool
- 说明书位置：references/weather-tool.md
- 备注：使用 reqwest + clap，编译产物 2.3 MB
```

## 四、锻造规范（forge-spec.md）

锻造规范是天工的**灵魂文件**。每次开启锻造时，天工将锻造规范和锻造令一起喂给 Coding Agent，Coding Agent 按照规范完成全部工作。

天工自身不实现编译、交付、写说明书——这些逻辑全写在锻造规范里，由 Coding Agent 执行。

### 4.1 锻造规范内容

```markdown
# 天工锻造规范

## 你的角色

你是天工的执行者，负责按照锻造令的要求，在 Rust 工作空间中编写 CLI 工具，
并完成编译、测试、交付的全流程。

## 环境说明

- Rust 工作空间：/workspace/（你可以修改这里的任何文件）
- 交付目录：/shared/skills/TianGongToolList/
  - 二进制放到：/shared/skills/TianGongToolList/scripts/
  - 说明书放到：/shared/skills/TianGongToolList/references/
  - 工具清单：/shared/skills/TianGongToolList/SKILL.md

## 锻造流程

按以下步骤完成锻造：

### 第一步：理解需求
阅读锻造令，明确工具功能、参数、输入输出。

### 第二步：编写 Rust CLI
在 /workspace/ 下编写 Rust CLI 工具：
- 使用 `clap` 框架处理命令行参数（derive 模式）
- 必须实现 `--help` 参数
- 默认输出格式为 JSON（方便程序解析）
- 使用 `anyhow` 处理错误
- 退出码规范：0=成功，非0=失败，stderr 输出错误信息

### 第三步：编译
执行 `cargo build --release`，确保编译通过。
如果编译失败，阅读错误信息并修复，直到编译成功。

### 第四步：交付二进制
将编译产物复制到交付目录：
```bash
cp /workspace/target/release/{tool-name} /shared/skills/TianGongToolList/scripts/
chmod +x /shared/skills/TianGongToolList/scripts/{tool-name}
```

### 第五步：编写说明书
在 /shared/skills/TianGongToolList/references/ 下创建 {tool-name}.md。
说明书格式：

```markdown
# {tool-name} 使用说明书

## 功能
（简述工具功能）

## 用法
（命令行用法）

## 参数
| 参数 | 必填 | 说明 | 示例 |
|------|------|------|------|

## 输出示例
（JSON 输出示例）

## 错误码
| 退出码 | 含义 |
|--------|------|

## 注意事项
（环境变量、网络依赖等）
```

### 第六步：更新工具清单
编辑 /shared/skills/TianGongToolList/SKILL.md：
- 在"已锻造"表格追加一行：`| {工具名} | {简介} | references/{工具名}.md | scripts/{工具名} |`
- 如果"锻造中"表格有该工具的记录，删除它

### 第七步：验证
执行交付的二进制，传入 --help，确认输出正常。

## 常用 Rust 依赖

工作空间已预热以下 crate（编译快）：
- `clap`（derive 模式）— 命令行参数
- `serde` + `serde_json` — JSON 序列化
- `reqwest`（blocking 或 tokio） — HTTP 请求
- `tokio` — 异步运行时
- `anyhow` — 错误处理

## 注意事项

- 每次锻造一个工具，不要修改其他已有工具
- 如果锻造令中提到外部 API，需要环境变量传入 API Key，不要硬编码
- 工具二进制必须能独立运行，不依赖工作空间中的任何文件
```

## 五、天工调度器实现

天工的 Python 代码极简，只有三个文件。

### 5.1 核心引擎（engine.py）

```python
import asyncio
import shutil
from pathlib import Path
from datetime import datetime


class TianGongEngine:
    """
    天工核心引擎

    职责极简：
    1. 定时巡查 orders/pending/ 目录
    2. 发现锻造令 → 启动 Coding Agent 执行
    3. 监控结果 → 归档锻造令

    所有锻造逻辑（写代码、编译、交付、写说明书）
    全部由 Coding Agent 按照锻造规范自行完成。
    """

    def __init__(self, config: dict):
        self.shared_dir = Path(config["shared_dir"])
        self.workspace = config["workspace_dir"]
        self.poll_interval = config.get("poll_interval", 900)

        self.orders_dir = self.shared_dir / "tiangong" / "orders"
        self.forge_spec_path = self.shared_dir / "tiangong" / "forge-spec.md"

        # 根据配置创建对应的 Coding Agent 适配器
        from .adapters import CodingAgentAdapter
        self.agent = CodingAgentAdapter.create(config["agent"])

    async def run(self):
        """主循环：定时巡查"""
        self._ensure_dirs()
        while True:
            await self._patrol()
            await asyncio.sleep(self.poll_interval)

    async def _patrol(self):
        """巡查一轮 pending 目录"""
        pending_dir = self.orders_dir / "pending"
        orders = sorted(pending_dir.glob("*.md"))

        if not orders:
            return

        for order_file in orders:
            await self._forge(order_file)

    async def _forge(self, order_file: Path):
        """处理一个锻造令"""
        # 移到 processing
        proc_file = self._move_order(order_file, "processing")
        order_content = proc_file.read_text(encoding="utf-8")
        forge_spec = self.forge_spec_path.read_text(encoding="utf-8")

        # 拼接完整 prompt：锻造规范 + 本次锻造令
        prompt = (
            f"{forge_spec}\n\n"
            f"---\n\n"
            f"# 本次锻造令\n\n"
            f"{order_content}"
        )

        # 启动 Coding Agent
        result = await self.agent.run(prompt)

        # 归档
        self._archive_order(proc_file, result)

    def _move_order(self, order_file: Path, target: str) -> Path:
        """移动锻造令到指定子目录"""
        target_dir = self.orders_dir / target
        target_dir.mkdir(parents=True, exist_ok=True)
        dest = target_dir / order_file.name
        shutil.move(str(order_file), str(dest))
        return dest

    def _archive_order(self, proc_file: Path, result):
        """将锻造令归档到 done/，追加结果记录"""
        now = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
        status = "成功" if result.success else "失败"

        appendix = (
            f"\n\n---\n\n"
            f"## 锻造结果\n"
            f"- 状态：{status}\n"
            f"- 完成时间：{now}\n"
        )
        if not result.success:
            appendix += f"- 失败原因：{result.error}\n"
        if result.output:
            appendix += f"- Agent 输出摘要：{result.output[:500]}\n"

        with open(proc_file, "a", encoding="utf-8") as f:
            f.write(appendix)

        self._move_order(proc_file, "done")

    def _ensure_dirs(self):
        """确保目录结构存在"""
        for sub in ["pending", "processing", "done"]:
            (self.orders_dir / sub).mkdir(parents=True, exist_ok=True)
```

### 5.2 Coding Agent 适配器（adapters.py）

适配器模式解耦天工与具体 CLI 工具。切换 Coding Agent 只需改配置，不改代码。

```python
import asyncio
import json
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional


@dataclass
class AgentResult:
    """Coding Agent 执行结果"""
    success: bool
    output: str = ""
    session_id: Optional[str] = None
    error: Optional[str] = None


class CodingAgentAdapter(ABC):
    """
    Coding Agent 适配器基类

    每种 CLI 工具实现一个子类。
    天工通过适配器统一调用，不关心具体是哪个 CLI。

    接入新的 Coding Agent CLI 时，需要：
    1. 继承此基类
    2. 实现 run() 和 resume() 方法
    3. 在 create() 工厂方法中注册
    4. 查阅该 CLI 的非交互模式文档，确认：
       - 非交互执行命令及参数
       - session 恢复命令及参数
       - 输出格式（JSON / 纯文本）
       - 退出码规范
    """

    @abstractmethod
    async def run(self, prompt: str) -> AgentResult:
        """首次执行任务"""
        pass

    @abstractmethod
    async def resume(self, message: str) -> AgentResult:
        """恢复 session 继续执行（多轮交互场景）"""
        pass

    @staticmethod
    def create(config: dict) -> "CodingAgentAdapter":
        """工厂方法"""
        agent_type = config["type"]
        adapters = {
            "claude_code": ClaudeCodeAdapter,
            "codex": CodexAdapter,
            # "kimi": KimiAdapter,
            # "opencode": OpenCodeAdapter,
        }
        cls = adapters.get(agent_type)
        if not cls:
            raise ValueError(
                f"不支持的 Agent 类型: {agent_type}，"
                f"可选: {list(adapters.keys())}"
            )
        return cls(config)

    async def _exec(self, cmd: list, cwd: str) -> tuple[int, str, str]:
        """通用子进程执行"""
        proc = await asyncio.create_subprocess_exec(
            *cmd,
            cwd=cwd,
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE,
        )
        stdout, stderr = await proc.communicate()
        return proc.returncode, stdout.decode(), stderr.decode()


class ClaudeCodeAdapter(CodingAgentAdapter):
    """
    Claude Code CLI 适配器

    文档：https://code.claude.com/docs/en/cli-reference
    非交互模式：claude -p "prompt"
    输出格式：--output-format json（含 session_id）
    session 恢复：claude --continue -p "follow-up"
    """

    def __init__(self, config: dict):
        self.cwd = config["workspace_dir"]

    async def run(self, prompt: str) -> AgentResult:
        code, stdout, stderr = await self._exec(
            ["claude", "-p", "--output-format", "json", prompt],
            cwd=self.cwd,
        )
        if code == 0:
            try:
                data = json.loads(stdout)
                return AgentResult(
                    success=True,
                    output=data.get("result", stdout),
                    session_id=data.get("session_id"),
                )
            except json.JSONDecodeError:
                return AgentResult(success=True, output=stdout)
        return AgentResult(success=False, error=stderr or stdout)

    async def resume(self, message: str) -> AgentResult:
        code, stdout, stderr = await self._exec(
            ["claude", "--continue", "-p", "--output-format", "json", message],
            cwd=self.cwd,
        )
        if code == 0:
            try:
                data = json.loads(stdout)
                return AgentResult(
                    success=True,
                    output=data.get("result", stdout),
                    session_id=data.get("session_id"),
                )
            except json.JSONDecodeError:
                return AgentResult(success=True, output=stdout)
        return AgentResult(success=False, error=stderr or stdout)


class CodexAdapter(CodingAgentAdapter):
    """
    OpenAI Codex CLI 适配器

    文档：https://developers.openai.com/codex/cli/reference/
    非交互模式：codex exec "prompt"
    选项：--full-auto --json
    session 恢复：codex exec resume --last "follow-up"
    """

    def __init__(self, config: dict):
        self.cwd = config["workspace_dir"]

    async def run(self, prompt: str) -> AgentResult:
        code, stdout, stderr = await self._exec(
            ["codex", "exec", "--full-auto", "--json", prompt],
            cwd=self.cwd,
        )
        if code == 0:
            return AgentResult(success=True, output=stdout)
        return AgentResult(success=False, error=stderr or stdout)

    async def resume(self, message: str) -> AgentResult:
        code, stdout, stderr = await self._exec(
            ["codex", "exec", "resume", "--last", message],
            cwd=self.cwd,
        )
        if code == 0:
            return AgentResult(success=True, output=stdout)
        return AgentResult(success=False, error=stderr or stdout)
```

### 5.3 入口（main.py）

import asyncio
import yaml
from pathlib import Path
from .engine import TianGongEngine


def load_config() -> dict:

RUN cargo init && \

reqwest = { version = "0.12", features = ["json", "blocking"] }
tokio = { version = "1", features = ["full"] }
anyhow = "1"
TOML
RUN cargo build --release 2>&1 && rm -rf /tmp/warmup/src /tmp/warmup/Cargo.toml
# 保留 /tmp/warmup/target 中的编译缓存
# 后续在 /workspace 中通过 CARGO_TARGET_DIR 复用

# ── Rust 工作空间 ──
WORKDIR /workspace
RUN cargo init --name tiangong-tool

# ── 天工调度器 ──
COPY docker/tiangong/requirements.txt /app/
RUN pip3 install --break-system-packages -r /app/requirements.txt

COPY src/tiangong/ /app/tiangong/

# ── 共享卷挂载点 ──
VOLUME ["/shared"]

WORKDIR /app
CMD ["python3", "-m", "tiangong.main"]
```

### 7.2 预热机制说明

预热的目的是让第一次锻造就能快速编译：

```
不预热：
  Coding Agent 写完代码 → cargo build
  → 下载 clap, serde, reqwest... → 编译所有依赖
  → 2-5 分钟

预热后：
  Coding Agent 写完代码 → cargo build
  → 依赖已编译好 → 只编译新写的代码
  → 10-30 秒
```

预热在 Docker 镜像构建时完成，只做一次。

## 八、如意侧实现

### 8.1 tiangong_evolve 工具

如意侧的 `tiangong_evolve` 工具实现非常简单——就是写一个锻造令文件：

```python
from datetime import datetime
from pathlib import Path


class TianGongEvolveTool:
    """
    天工锻造请求工具

    如意调用此工具时，生成一份锻造令写入 orders/pending/ 目录。
    天工定时巡查后会自动开始锻造。
    """

    name = "tiangong_evolve"
    description = (
        "向天工下达锻造令，请求锻造新的 CLI 工具。"
        "天工会在下次巡查时开始锻造（默认每 15 分钟巡查一次）。"
    )

    def __init__(self, orders_dir: str):
        self.orders_dir = Path(orders_dir)

    async def execute(self, task: str) -> str:
        tool_name = self._extract_tool_name(task)
        now = datetime.now()
        filename = f"{now.strftime('%Y%m%d-%H%M%S')}-{tool_name}.md"

        order_content = self._build_order(tool_name, task, now)

        pending_dir = self.orders_dir / "pending"
        pending_dir.mkdir(parents=True, exist_ok=True)
        order_path = pending_dir / filename
        order_path.write_text(order_content, encoding="utf-8")

        return (
            f"锻造令已下达：{filename}\n"
            f"天工将在下次巡查时开始锻造 {tool_name}。"
        )

    def _extract_tool_name(self, task: str) -> str:
        """从任务描述中提取工具名（简单实现，可由 LLM 生成）"""
        # 实际实现中可以让 Agent 在 tool_call 参数里直接指定工具名
        return "new-tool"

    def _build_order(self, tool_name: str, task: str, now: datetime) -> str:
        return (
            f"# 锻造令：{tool_name}\n\n"
            f"## 需求描述\n\n{task}\n\n"
            f"## 元信息\n"
            f"- 创建时间：{now.strftime('%Y-%m-%d %H:%M:%S')}\n"
            f"- 请求方：如意\n"
            f"- 优先级：normal\n"
        )
```

### 8.2 如意怎么知道工具锻造好了

两条路径，可以共存：

**被动发现：** 用户下次提问时，如意触发 TianGongToolList Skill，重新读取 SKILL.md，发现新增了工具，自然就能用了。

**主动通知（依赖心跳机制）：** 如意心跳时可以检查 `orders/done/` 目录，发现有新完成的锻造令，主动通知用户：

```python
async def heartbeat_check_forge_results(self):
    """心跳时检查锻造结果（可选）"""
    done_dir = self.orders_dir / "done"
    for order_file in done_dir.glob("*.md"):
        content = order_file.read_text()
        if "状态：成功" in content and not self._already_notified(order_file):
            await self.notify_user(f"天工锻造完成！新工具已就绪。")
            self._mark_notified(order_file)
```

## 十、Coding Agent Session 支持

天工可能需要与 Coding Agent 多轮交互（例如编译失败后要求修复），各 CLI 的 session 支持情况：

### Claude Code

```bash
# 首次执行
claude -p --output-format json "根据锻造规范和锻造令完成锻造"
# 输出含 session_id

# 恢复 session（多轮）
claude --continue -p --output-format json "编译失败了，请修复以下错误..."
# --continue 恢复最近 session
```

- `--continue` 在非交互模式下稳定可用
- `--resume SESSION_ID` 目前在 Linux 上有 bug，建议用 `--continue`

### Codex

```bash
# 首次执行
codex exec --full-auto --json "根据锻造规范和锻造令完成锻造"
# session 自动保存到 ~/.codex/sessions/

# 恢复 session（多轮）
codex exec resume --last "编译失败了，请修复以下错误..."
# 或指定 session ID
codex exec resume <session-id> "请修复..."
```

- `resume --last` 和 `resume <session-id>` 均可用
- 保留完整对话历史和文件变更记录

### 适配器中的多轮交互

适配器基类提供 `resume()` 方法。天工可以在首次执行失败后调用 `resume()` 进行后续交互。但目前的极简实现中，**多轮交互逻辑也可以交给 Coding Agent 自己完成**——大多数 Coding Agent 在遇到编译失败时会自行重试，不需要天工介入。
