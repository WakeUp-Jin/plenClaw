# PineClaw 存储架构设计

## 一、核心理念

PineClaw 是长期个人助手，不存在"会话"概念——用户和助手之间是持续陪伴关系。记忆是连续的流，"遗忘"通过压缩 + 归档实现，而非切换会话。

所有用户资产（配置、记忆、技能）统一放在一个用户级目录 `~/.pineclaw/` 下，可以：
- 随时整体迁移到其他机器
- Docker 挂载一个目录即可持久化全部数据
- 首次启动自动创建目录结构和默认配置

## 二、目录结构

```
~/.pineclaw/                                 # 唯一用户级目录
├── config.json                              # 统一配置文件
├── skills/                                  # Skill 目录（扫描器自动发现）
│   └── other-skill/
│       └── SKILL.md
└── memory/                                  # 记忆系统（核心功能）
    ├── short_term/                          # 短期记忆
    │   ├── state.json                       # 活跃段落指针
    │   ├── 20260325_143022/                 # 已归档的时间段
    │   │   ├── history.jsonl                # 原始完整记录
    │   │   └── checkpoint.json              # 压缩检查点（摘要 + 保留行号）
    │   ├── 20260325_160000/                 # 当前活跃时间段
    │   │   └── history.jsonl                # 还没压缩，只有原始记录
    │   └── ...
    └── long_term/                           # 长期记忆（LLM 自由创建）
        ├── profile.md                       # 用户画像
        ├── preferences.md                   # 偏好设置
        └── ...
```

## 三、统一配置 `config.json`

位置：`~/.pineclaw/config.json`（项目根目录的旧 `config.json` 不再使用）。

### 配置结构

```json
{
  "app": {
    "log_level": "INFO"
  },
  "models": {
    "high": {
      "id": "kimi-k2.5",
      "name": "Kimi K2.5",
      "provider": "kimi",
      "api_key": "${KIMI_API_KEY}",
      "base_url": "https://api.moonshot.cn/v1",
      "reasoning": false,
      "context_window": 131072,
      "max_tokens": 4096,
      "temperature": 1.0,
      "cost": { "input": 2.0, "output": 8.0 }
    },
    "medium": { "..." },
    "low": { "..." }
  },
  "memory": {
    "short_term": {
      "compression_threshold": 0.8,
      "compress_keep_ratio": 0.3
    }
  },
  "retry": { "max_retries": 3, "base_delay": 1.0, "max_delay": 30.0 },
  "feishu": { "..." }
}
```

### ModelConfig 关键字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `str` | 模型 ID，传给 API |
| `name` | `str` | 人类可读名称 |
| `provider` | `str` | 供应商标识 |
| `reasoning` | `bool` | 是否支持 thinking/推理 |
| `context_window` | `int` | 模型上下文窗口大小（token） |
| `max_tokens` | `int` | 单次生成的最大 token |
| `cost.input` | `float` | 输入价格，**元 / 百万 token** |
| `cost.output` | `float` | 输出价格，**元 / 百万 token** |

### cost 计算

```python
model_cfg = settings.get_model_config("high")
# 500 tokens 的输入消息
cost_cny = model_cfg.calc_cost(500, direction="input")
# = 500 * 2.0 / 1_000_000 = 0.001 元
```

### 压缩判断

```python
# CompressionConfig.context_window 来自 model 配置
# 触发阈值 = context_window × compression_threshold
config = CompressionConfig(
    context_window=model_cfg.context_window,   # 131072
    compression_threshold=settings.compression_threshold,  # 0.8
)
# 当 token 用量 >= 131072 * 0.8 = 104857 时触发压缩
```

## 四、ContextItem 数据结构

`ContextItem` 是上下文系统的核心数据单元，贯穿整个流程。

### 两套序列化

| 方法 | 用途 | 包含字段 |
|------|------|----------|
| `to_message()` | 发送给 LLM API | role, content, tool_calls, tool_call_id, name |
| `to_dict()` | 写入 JSONL 持久化 | 全部字段（含 thinking, cost, token_estimate 等） |

### 新增字段

```python
@dataclass
class ContextItem:
    # ... 原有字段 ...

    # 模型思考链（不发送给 LLM）
    thinking: str | None = None
    thinking_token_estimate: int = 0

    # 花费（人民币元）
    cost: float = 0.0
```

- **`thinking`**：模型推理链文本（如 DeepSeek-R1 / Claude 的 thinking 输出），存储但不发送给 API
- **`cost`**：该消息的花费（人民币元），user 消息按 input 价格计算，assistant 消息按 output 价格计算

## 五、短期记忆结构

### 文件格式

**`history.jsonl`** —— 每行一个 `ContextItem.to_dict()` 的 JSON：

```jsonl
{"role":"user","content":"你好","source":"user","priority":3,"created_at":1711000000.0,"token_estimate":5,"cost":0.00001,...}
{"role":"assistant","content":"你好！有什么可以帮你的？","thinking":"用户在打招呼...","cost":0.00008,...}
```

**`checkpoint.json`** —— 压缩后生成：

```json
{
  "summary": "用户询问了 Python 装饰器的用法，助手给出了详细解释和代码示例...",
  "compressed_lines": 70,
  "kept_from_line": 71,
  "total_lines": 100,
  "archive_hint": "完整记录保存在 history.jsonl 前 70 行",
  "created_at": "2026-03-25T16:00:00"
}
```

**`state.json`** —— 短期记忆全局状态：

```json
{
  "active_folder": "20260325_160000",
  "created_at": "2026-03-25T16:00:00"
}
```

### 压缩机制

触发条件：`当前 token 总量 ≥ context_window × compression_threshold`

压缩流程：
1. 取前 70% 的消息，用 LLM 生成摘要
2. 摘要 + 保留信息写入 `checkpoint.json`
3. 保留后 30% 的原始消息在内存中
4. 原始 `history.jsonl` 保留不删（归档）
5. 新的活跃记录写入新时间段文件夹

加载流程：
1. 读取 `state.json` 找到活跃文件夹
2. 如有 `checkpoint.json`，加载摘要作为上下文开头
3. 加载 `history.jsonl` 中 `kept_from_line` 之后的记录

## 六、长期记忆

路径：`~/.pineclaw/memory/long_term/`

LLM 自由创建和维护 Markdown 文件，例如：
- `profile.md` —— 用户画像（姓名、职业、背景）
- `preferences.md` —— 使用偏好（编程语言、框架偏好）
- `project_notes.md` —— 项目备忘

长期记忆通过工具（`memory_read` / `memory_write`）操作，由 `LocalMemoryStore` 提供持久化。

services:

    volumes:
      - ~/.pineclaw:/root/.pineclaw

## 八、ShortMemoryStore 存储类

`src/storage/short_memory_store.py` 替代原有的 `ConversationStore`，实现 `IContextStorage` 接口。

### 核心方法

| 方法 | 作用 |
|------|------|
| `append(dict)` | 向活跃段的 `history.jsonl` 追加一行 |
| `load_all()` | 加载活跃段全部记录 |
| `load_from_line(n)` | 从第 n 行开始加载（压缩恢复用） |
| `save_checkpoint(summary, line)` | 写入 `checkpoint.json` |
| `load_checkpoint()` | 读取 checkpoint |
| `rotate()` | 归档当前段，创建新段，更新 `state.json` |
| `list_segments()` | 列出所有时间段文件夹（工具用） |

### 压缩后的 rotate 流程

```
compress() 触发
  ├─ LLM 生成摘要
  ├─ save_checkpoint() 写入当前段的 checkpoint.json
  ├─ rotate() 创建新的时间段文件夹
  ├─ 保留的 30% 消息写入新段的 history.jsonl
  └─ state.json 指向新段
```

### IContextStorage 接口变更

去掉了 `new_conversation()` 方法（不再有"会话"概念），其余方法签名不变。

## 九、代码影响总结

| 文件 | 改动 |
|------|------|
| `src/config/settings.py` | 新增 `get_pineclaw_home()`、`ensure_pineclaw_dirs()`、`ModelConfig`（含 `context_window` / `cost`）、`CostConfig` |
