# 助手Agent记忆模块 - 完整需求文档

---

## 一、模块概述

### 1.1 模块定位

记忆模块是助手Agent的核心基础设施，赋予Agent跨会话、跨时间的持续记忆能力。它作为Agent Skill体系中的一个特殊Skill存在，但具有与普通Skill不同的加载机制——**每次交互必须自动加载**。

### 1.2 设计理念

模仿人类记忆的运作方式：
- **长期记忆**：如同人类的深层记忆，存储稳定的、经过沉淀的重要信息
- **短期记忆**：如同人类的近期经历，保留最近的交互细节，会逐渐遗忘（压缩）
- **遗忘机制**：人类不会记住所有细节，Agent也不应该。通过分层压缩实现自然的"遗忘"

### 1.3 核心目标

1. Agent能记住用户是谁、喜欢什么、要求过什么
2. Agent能回忆起近期的交互内容，保持对话连贯性
3. 记忆系统能在有限的上下文窗口内高效运作
4. 记忆内容能随时间自然压缩，不会无限膨胀
5. 重要信息永远不会丢失，可通过溯源链接找回

## 三、SKILL.md 定义

### 3.1 元信息

```yaml
---
name: "用户记忆"
type: memory
always_load_content: true
description: "用户的记忆模块。包含用户指令、偏好、交互历史。标记为always_load_content，每次交互自动加载正文。"
---
```

**⚠️ 重点说明：`always_load_content: true`**

这个字段是记忆Skill区别于所有其他Skill的关键标志。正常Skill的正文加载是Agent在第一层扫描元信息后，根据用户输入按需决定是否进入第二层加载正文。但记忆Skill必须每次都加载正文，因为正文中包含了长期记忆的文件索引和短期记忆的加载策略，Agent需要这些信息才能正确初始化记忆状态。

**在Agent主系统提示词中必须配合声明：**
```markdown
## Skill加载规则
- 扫描所有Skill的元信息时，如果发现 `always_load_content: true`，
  必须立即读取该Skill的完整正文内容，不需要等待用户输入判断。
- 此类Skill的正文加载优先级高于所有其他操作。
```

### 3.2 正文内容

SKILL.md的正文是Agent每次交互的第一份"记忆地图"，它告诉Agent：有哪些长期记忆文件、哪些必须加载、哪些按需加载、短期记忆怎么读取。

```markdown
# 用户记忆模块

## 长期记忆文件索引

### 必加载文件（每次交互都需要读取）
| 文件 | 说明 | 优先级 |
|------|------|--------|
| long_term/user_instructions.md | 用户的明确指令和规则 | 🔴 最高 |
| long_term/user_profile.md | 用户基本信息和画像 | 🔴 最高 |
| long_term/corrections.md | 用户纠正过的错误记录 | 🔴 最高 |

### 按需加载文件（根据对话内容判断是否需要）
| 文件 | 说明 | 加载条件 |
|------|------|----------|
| long_term/project_context.md | 项目相关上下文 | 用户讨论项目或技术时 |
| long_term/facts_and_decisions.md | 重要事实和决策 | 用户涉及决策或引用历史时 |
| long_term/topics_and_interests.md | 话题和兴趣图谱 | 用户闲聊或跨领域话题时 |
| long_term/interaction_patterns.md | 交互模式和偏好 | 需要调整交互方式时 |

## 短期记忆加载策略

### 存储位置
short_term/ 目录，按月份子文件夹组织。

### 加载规则
1. 从当前月份文件夹开始，按时间倒序扫描
2. 如果某个日期范围存在周摘要（.summary.md），加载摘要，跳过对应日记录
3. 如果某个月份存在月摘要，加载月摘要，跳过该月份文件夹的其他内容
4. 按Token预算加载，预算用完即停止
5. 今天的日记录始终加载（当前正在进行的交互记录）

### 压缩触发
当已用容量 ≥ 短期记忆总容量的 85% 时触发压缩。
详见压缩策略部分。

## 记忆更新规则

### 短期记忆更新
- 每轮对话结束后，将交互摘要追加到今天的日记录文件中

### 长期记忆更新
- 在交互过程中识别到需要长期记忆的信息时，更新对应的长期记忆文件
- 长期记忆的更新条件见各文件的具体说明
```

## 四、长期记忆详细设计

### 4.1 设计原则

- 长期记忆文件数量**有限且稳定**，不会随时间无限增长
- 每个文件有**明确的职责边界**，不会出现信息重叠
- 文件通过**更新**而非**追加**的方式维护，保持文件体积可控
- 所有文件都有明确的**加载条件**（必加载 / 按需加载）

### 4.2 必加载文件详细说明

#### 4.2.1 user_instructions.md（用户指令集）

**职责**：记录用户**明确要求Agent遵守**的指令和规则。

**⚠️ 重点说明：这是优先级最高的记忆文件。**

用户说的"以后都要这样做"、"记住不要那样做"这类内容，必须写入此文件。Agent在每次交互中都必须遵守这些指令，违反这些指令等同于违反系统提示词。

**文件格式**：

```markdown
# 用户指令集

## 回复风格指令
- 回复使用中文
- 代码注释使用中文
- 不要过度使用emoji
- 解释技术概念时用类比，不要太学术化

## 工作流程指令
- 修改代码前先说明要改什么，等我确认后再改
- 每次只改一个文件，不要一次改多个
- 不确定的时候要问我，不要自己猜

## 禁止事项
- 不要在回复开头说"好的"
- 不要自作主张删除我的注释
- 不要使用 var 声明变量，统一用 const/let

## 元信息
- 最后更新：2025-01-16
- 条目数量：10条
```

**更新触发条件**：
- 用户使用"以后..."、"记住..."、"每次都要..."等祈使句
- 用户对Agent的行为进行纠正并要求以后避免
- 用户明确表示"这是一条指令"或类似意图

**更新方式**：追加新指令条目，或修改已有条目。如果新指令与旧指令矛盾，以新指令为准，移除旧指令。

---

#### 4.2.2 user_profile.md（用户画像）

**职责**：记录用户的**基本信息和个人特征**，帮助Agent建立对用户的整体认知。

**文件格式**：

```markdown
# 用户画像

## 基本信息
- 称呼：小明
- 职业：前端开发工程师
- 技术栈：React, TypeScript, Node.js
- 经验水平：3年工作经验，中高级

## 性格与沟通特点
- 喜欢直接了当，不喜欢冗长的解释
- 技术讨论时倾向于先看代码再听解释
- 遇到bug时会比较急躁，这时需要简洁明了的回复

## 工作环境
- 操作系统：macOS
- 编辑器：VS Code
- 主力浏览器：Chrome
- 包管理器：pnpm

## 当前关注点
- 正在学习系统设计
- 对AI Agent开发很感兴趣
- 最近在重构公司的后台管理系统

## 元信息
- 创建时间：2025-01-01
- 最后更新：2025-01-16
```

**更新触发条件**：
- 用户主动分享个人信息
- 通过交互推断出新的用户特征（需要高置信度）
- 用户纠正了已有的画像信息

**⚠️ 重要原则**：推断出的信息需要标注`[推断]`标签，与用户明确告知的信息区分。不确定的推断不写入。

#### 4.2.3 corrections.md（纠错记录）

**职责**：记录用户**纠正过Agent的错误**，防止同类错误再次发生。

**⚠️ 重点说明：这是Agent自我改进的核心文件。**

当用户说"不对"、"你搞错了"、"不是这样的"时，Agent不仅要当场改正，还要把这次纠错记录下来，确保以后不再犯同样的错误。

**文件格式**：

```markdown
# 纠错记录

## 技术类纠错

### C001 - React状态更新
- 日期：2025-01-10
- 错误：建议在useEffect中直接修改state导致无限循环
- 正确做法：使用函数式更新或添加正确的依赖项
- 用户原话："这样会死循环的，你应该知道的"

### C002 - TypeScript类型
- 日期：2025-01-12
- 错误：建议使用any类型来解决类型冲突
- 正确做法：使用泛型或类型守卫来正确处理
- 用户原话："别用any，我之前说过的"

## 偏好类纠错

### P001 - 命名风格
- 日期：2025-01-08
- 错误：使用camelCase命名CSS类
- 正确做法：使用kebab-case命名CSS类
- 用户原话："CSS类名用横杠连接"

## 元信息
- 最后更新：2025-01-12
- 总条目数：3条
```

**更新触发条件**：
- 用户纠正了Agent的任何错误
- 用户表达不满并指出问题所在
- 同一类错误出现第二次时，需要升级该条目的重要性标注

**更新方式**：追加新条目。不删除旧条目（除非用户明确说"之前那个纠正不算了"）。

## 五、短期记忆详细设计

### 5.1 设计原则

- 短期记忆是Agent对**近期交互细节的保持**
- 所有短期记忆共享一个**Token预算**，预算由上下文窗口决定
- 越近的记忆越完整，越远的记忆越精炼（通过压缩实现）
- 原始文件**永远保留**在磁盘上，压缩只影响加载策略
- 当前对话和历史记录共享短期记忆空间，这是自然的

### 5.2 日记录文件格式

每天一个文件，记录当天所有交互的摘要。

**文件命名**：`YYYY-MM-DD.md`

**文件格式**：

```markdown
# 2025-01-16 交互记录

## 10:30 - 讨论记忆模块设计
- 用户提出了助手Agent记忆模块的需求
- 讨论了长期记忆和短期记忆的架构设计
- 确定了7个长期记忆文件的分类
- 🔴 用户要求：长期记忆文件不要超过7个
- 溯源：本次对话完整记录

## 14:00 - React组件优化
- 用户分享了一个性能问题：列表渲染卡顿
- 分析原因：缺少key导致全量重渲染
- 解决方案：添加唯一key + React.memo
- 🟡 用户学到了：React.memo的使用场景

## 16:30 - 闲聊
- 用户提到周末打算去爬山
- 🟢 用户兴趣：户外运动
```

**⚠️ 重点说明：每条记录的重要性标注**

- 🔴 **关键**：影响长期记忆的信息、用户的明确指令、重要决策
- 🟡 **一般**：有参考价值的技术讨论、学习收获
- 🟢 **次要**：闲聊、一次性问题、情绪性内容

重要性标注在压缩时非常关键：周摘要保留 🔴🟡🟢，月摘要只保留 🔴🟡，年摘要只保留 🔴。

### 5.3 写入策略

```markdown
## 日记录写入规则

### 写入时机
- 每轮对话结束后（Agent回复完成后），将本轮交互的摘要追加到当天的日记录文件

### 写入内容（非原文记录，而是摘要）
- 交互的主题和时间
- 关键信息点（用户说了什么重要的、做了什么决策）
- 结论和结果
- 重要性标注

### 不写入的内容
- 完整的对话原文（太占空间）
- Agent的详细回复内容（只记结论）
- 纯粹的代码块（只记"写了什么功能"）

### 示例：一轮对话的摘要写入
用户问："帮我写一个防抖函数"
Agent回复了一段代码并解释

写入日记录的内容：
  ## 14:30 - 实现防抖函数
  - 用户需要一个防抖函数
  - 提供了TypeScript版本的debounce实现
  - 🟢 一次性技术问题
```

## 六、短期记忆加载策略

### 6.1 Token预算计算

```
短期记忆总容量 = 模型上下文窗口 - 固定开销

其中固定开销包括：
  · 系统提示词
  · 其他Skill加载
  · 记忆Skill的SKILL.md正文
  · 长期记忆必加载文件
  · 长期记忆按需加载文件（如果本次有加载的话）

示例：
  模型上下文窗口：128K tokens
  固定开销：约 15K tokens
  短期记忆总容量：约 113K tokens
```

**⚠️ 重点说明**

短期记忆总容量包含了**当前正在进行的对话**。也就是说，随着当前对话的增长，留给历史短期记忆的空间会被挤压。这是设计预期内的行为——正如人类在深度交谈时会暂时"忘记"更久远的细节一样。

### 6.2 初始加载流程

每次新交互开始时，Agent按以下流程加载短期记忆：

```
步骤1：计算短期记忆总容量
        ↓
步骤2：设置初始加载预算 = 总容量 × 70%
       （预留30%给当前对话的增长空间）
        ↓
步骤3：从当前月份文件夹开始，按时间倒序扫描
        ↓
步骤4：对于当前月份内的文件，按以下优先级加载：
       a. 今天的日记录 → 必定加载
       b. 从昨天往回逐天检查：
          - 该天是否被某个周摘要覆盖？
            YES → 加载周摘要（如果还没加载过），跳过该天的日记录
            NO  → 加载该天的日记录
       c. 每加载一个文件，更新剩余预算
       d. 预算用完 → 停止加载
        ↓
步骤5：如果当前月份加载完毕还有预算，进入上一个月：
       a. 该月有月摘要？
          YES → 加载月摘要，跳过该月的所有其他文件
          NO  → 进入该月文件夹，重复步骤4的逻辑
       b. 预算用完 → 停止加载
        ↓
步骤6：继续往更早的月份扫描，直到预算用完
```

**可视化示例**：

```
假设今天是 2025-01-16，短期记忆目录状态如下：

2025-01/
  2025-01-16.md          ← ① 加载（今天，必加载）
  2025-01-15.md          ← ② 加载（日记录）
  2025-01-14.md          ← ③ 加载（日记录）
  2025-01-13.md          ← ④ 加载（日记录）
  2025-01-12.md          ← ⑤ 加载（日记录）
  2025-01-11.md          ← ⑥ 加载（日记录）
  2025-01-10.md          ← ⑦ 加载（日记录）
  2025-01-09.md          ← ⑧ 加载（日记录）
  2025-01-08.md          ←    跳过！被周摘要覆盖
  week_01-01_to_01-07.summary.md  ← ⑨ 加载（周摘要）
  2025-01-07.md          ←    跳过！被周摘要覆盖
  2025-01-06.md          ←    跳过！
  ...
  2025-01-01.md          ←    跳过！

2024-12/
  month_2024-12.summary.md  ← ⑩ 加载（月摘要） 预算差不多了
  （该月其他文件全部跳过）

2024-11/
  month_2024-11.summary.md  ← 预算用完，停止加载
```

### 6.3 初始加载预算说明

初始加载时预留30%给当前对话，这个30%不是浪费：

```
交互开始时：
  历史短期记忆占 70%
  当前对话占 0%

交互进行中：
  历史短期记忆占 70%
  当前对话不断增长...占 15%...25%...

触发压缩线（85%）：
  历史短期记忆占 70%
  当前对话占 15%+
  → 触发压缩，释放历史记忆空间

压缩后：
  历史短期记忆占 55%（压缩释放了15%）
  当前对话继续增长...
  → 可以再次触发压缩
```

**这样的设计确保了**：
- 对话开始时有足够的历史上下文
- 对话过程中不会因为历史记忆太多而立刻触发压缩
- 当对话确实很长时，自然地压缩最旧的记忆来腾出空间

## 七、短期记忆压缩策略

### 7.1 压缩触发条件

```markdown
## 触发条件
当 已用容量 ≥ 短期记忆总容量 × 85% 时触发压缩。

已用容量 = 已加载的所有短期记忆文件Token + 当前对话已消耗的Token
```

### 7.2 压缩层级体系

```markdown
## 三层压缩层级

层级0：日记录（原始文件）
  ↓ 压缩（7天 → 1个摘要）
层级1：周摘要
  ↓ 压缩（4个周摘要 → 1个月摘要）
层级2：月摘要
  ↓ 压缩（多个月摘要 → 1个年摘要）（极端情况下才会触发）
层级3：年摘要
```

### 7.3 压缩执行流程

```
触发压缩！
    │
    ▼
当前已加载的短期记忆中，最早的内容是什么层级？
    │
    ├─── 存在未被压缩的日记录？
    │         │
    │        YES
    │         │
    │         ▼
    │    从最早的日记录开始，取最早的连续N天（目标7天，不足也可以）
    │    压缩为一个周摘要文件
    │    从已加载内容中移除这N天的日记录，替换为周摘要
    │    将周摘要文件写入磁盘
    │         │
    │         ▼
    │    释放的空间够吗？（已用容量 < 75%？）
    │         │
    │     YES: 压缩完成，继续对话
    │     NO:  继续压缩（回到起点，再看最早的内容）
    │
    ├─── 所有日记录都已压缩为周摘要，最早的是周摘要？
    │         │
    │        YES
    │         │
    │         ▼
    │    取最早的连续4个周摘要（不足4个也可以）
    │    压缩为一个月摘要文件
    │    从已加载内容中移除这些周摘要，替换为月摘要
    │    将月摘要文件写入磁盘
    │         │
    │         ▼
    │    释放的空间够吗？
    │     YES: 完成
    │     NO:  继续
    │
    └─── 最早的内容已经是月摘要？
              │
              ▼
         取最早的多个月摘要
         压缩为年摘要文件
         替换加载
```

### 7.4 压缩内容保留策略

不同层级的压缩保留不同重要性的信息：

```markdown
## 日记录 → 周摘要
- 保留：🔴 关键 + 🟡 一般 + 🟢 次要（全部保留，但语言更精炼）
- 压缩比：原文的 15%-25%
- 格式：按天列出要点

## 周摘要 → 月摘要
- 保留：🔴 关键 + 🟡 一般（丢弃 🟢 次要）
- 压缩比：周摘要总量的 20%-30%
- 格式：按周列出核心要点

## 月摘要 → 年摘要
- 保留：🔴 关键（只保留最关键的信息）
- 压缩比：月摘要总量的 20%-30%
- 格式：按月列出关键里程碑
```

### 7.5 摘要文件命名规范

```markdown
## 命名规范

周摘要：week_{起始日期}_to_{结束日期}.summary.md
  示例：week_01-01_to_01-07.summary.md

月摘要：month_{年-月}.summary.md
  示例：month_2024-12.summary.md

年摘要：year_{年}.summary.md
  示例：year_2024.summary.md

## 存放位置
- 周摘要放在对应月份的文件夹中
- 月摘要放在对应月份的文件夹中
- 年摘要放在 short_term/ 根目录
```

## 八、长期记忆更新机制

### 8.1 更新时机

长期记忆的更新不是定时任务，而是在交互过程中**实时识别和执行**的。

```markdown
## Agent在每轮对话中需要判断

1. 用户是否发出了新的指令？ → 更新 user_instructions.md
2. 用户是否透露了新的个人信息？ → 更新 user_profile.md
3. 用户是否纠正了我的错误？ → 更新 corrections.md
4. 用户是否做出了重要决策？ → 更新 facts_and_decisions.md
5. 用户是否展现了新的兴趣？ → 更新 topics_and_interests.md
6. 我的交互方式是否需要调整？ → 更新 interaction_patterns.md
7. 项目是否有新进展？ → 更新 project_context.md
```

### 8.2 更新原则

```markdown
## 更新规则

1. **原子性更新**：每次只更新一个文件中的一个或几个条目，不做大范围重写
2. **来源标注**：每条更新都标注来源日期或短期记忆引用
3. **冲突处理**：新信息与旧信息矛盾时，以新信息为准，旧信息标记为[已过时]或移除
4. **置信度控制**：推断出的信息标注[推断]，只有高置信度的推断才写入
5. **体积控制**：每个长期记忆文件控制在合理大小内（建议不超过3K tokens）
6. **告知用户**：更新长期记忆时，简要告知用户"我已经记住了..."，保持透明
```

### 8.3 长期记忆与短期记忆的关系

```
短期记忆（大量细节，会压缩）
    │
    │  沉淀
    │  ─────→  只有重要的、稳定的、反复出现的信息
    │           才会沉淀到长期记忆
    ▼
长期记忆（精炼信息，持久保存）

反向关系：
长期记忆 ──溯源链接──→ 短期记忆原文（可追溯到完整上下文）
```

## 九、压缩Agent设计

### 9.1 定位

压缩Agent是一个**专用的子Agent**，由主Agent在需要压缩短期记忆时调用。它不直接与用户交互。

### 9.2 压缩Agent系统提示词

```markdown
# 短期记忆压缩Agent

## 你的角色
你是一个记忆压缩专家。你的任务是将多个短期记忆记录压缩为精炼的摘要，
保留最重要的信息，让未来的Agent能通过摘要快速理解这段时间发生了什么。

## 输入
你会收到以下内容：
1. 压缩类型：日记录→周摘要 / 周摘要→月摘要 / 月摘要→年摘要
2. 需要压缩的源文件内容
3. 当前的长期记忆文件列表（用于判断哪些信息已经在长期记忆中，不需要在摘要中重复强调）

## 压缩原则

### 必须保留
1. 用户做出的明确决策和结论
2. 用户表达的偏好、需求、情感状态变化
3. 重要的事件和里程碑
4. 未解决的问题和待办事项
5. 用户要求记住的任何内容
6. 与长期记忆相关的线索（标注是否已同步到长期记忆）

### 可以丢弃
1. 具体的代码实现细节（除非是关键决策的直接体现）
2. 反复试错的中间过程（只保留最终结果）
3. 纯粹的寒暄和客套
4. Agent的详细解释内容（只保留结论）
5. 已经被完全解决且不会复现的一次性问题

### 重要性标注保留规则
- 日记录→周摘要：保留 🔴🟡🟢 全部
- 周摘要→月摘要：保留 🔴🟡，丢弃 🟢
- 月摘要→年摘要：只保留 🔴

## 输出格式
严格按照对应层级的摘要模板格式输出。
在末尾附上"本周/本月未解决的问题"和"需要关注的长期记忆更新"两个板块。
```

### 9.3 压缩执行方式

```markdown
## 压缩执行选项

### 方案一：同步执行（简单但影响响应）
- 主Agent检测到需要压缩时，暂停当前对话处理
- 调用压缩Agent执行压缩
- 压缩完成后，更新已加载的短期记忆
- 继续处理用户消息
- 缺点：用户会感到响应延迟

### 方案二：异步执行（推荐）
- 主Agent检测到需要压缩时，标记需要压缩的文件
- 在后台异步调用压缩Agent
- 当前对话继续进行（可能暂时略过一些历史上下文）
- 下次交互开始时，加载已完成的摘要文件
- 优点：不影响用户体验

### 方案三：会话间执行（最不影响体验）
- 当前对话中只标记"需要压缩"
- 在两次交互之间的空闲时间执行压缩
- 下次交互开始时，摘要文件已经就绪
- 优点：完全不影响当前对话
- 缺点：如果单次对话极长，可能来不及压缩
```

## 十一、边界情况处理

### 11.1 首次使用（全新用户）

```markdown
所有记忆文件都不存在。

处理方式：
1. SKILL.md正文加载后发现长期记忆文件不存在 → 跳过，不报错
2. 短期记忆目录为空 → 跳过加载
3. 在第一次交互中，根据用户的自我介绍或表现，开始创建各记忆文件
4. 可以主动友好地问用户一些基本信息（但不强制）
```

### 11.2 单日交互量极大

```markdown
如果用户一天内聊了极多内容，当天日记录文件可能非常大。

处理方式：
1. 日记录写入的是摘要而非原文，单日文件通常不会太大
2. 即使日记录较大，也会在下次交互开始时被正常加载（作为"今天"的记录）
3. 如果当天记录确实超出预算的40%，在当天对话结束后自动生成当天的摘要版
   （命名为 2025-01-16.summary.md，放在同目录下）
4. 次日加载时，发现同一天有 .summary.md 就加载摘要版
```

### 11.3 长时间不使用后恢复

```markdown
如果用户几个月没有使用，然后突然回来。

处理方式：
1. 长期记忆文件全部正常加载（它们一直在）
2. 短期记忆从最新的文件开始加载，可能直接加载到几个月前的月摘要
3. Agent可以说"好久不见"，基于长期记忆中的用户画像进行问候
4. 正常恢复交互
```

### 11.4 记忆文件损坏或缺失

```markdown
处理方式：
1. 任何记忆文件加载失败 → 跳过该文件，不影响其他文件加载
2. 在本次交互中尝试重建缺失的文件
3. 告知用户"我发现记忆中的某个文件似乎丢失了，我会在交互过程中重新建立"
```

### 11.5 不足7天的压缩

```markdown
如果最早的日记录不足7天（比如只有3天），仍然可以压缩。

处理方式：
1. 按实际天数压缩，命名体现实际范围
   例：week_01-05_to_01-07.summary.md（虽然只有3天，仍用week前缀保持一致）
2. 不必等凑够7天再压缩
```

## 十三、文档总结

| 维度 | 设计要点 |
|------|---------|
| **架构** | 长期记忆 + 短期记忆双层架构，通过SKILL.md作为入口 |
| **加载** | 记忆Skill通过`always_load_content: true`始终加载正文 |
| **长期记忆** | 7个文件，3个必加载+4个按需加载，通过更新维护 |
| **短期记忆** | 按月文件夹组织，日记录+周摘要+月摘要三层结构 |
| **加载策略** | 基于Token预算，从最近往回加载，有摘要加载摘要 |
| **压缩触发** | 已用容量≥85%时触发，压缩最旧的同层级内容 |
| **压缩层级** | 日记录→周摘要→月摘要→年摘要，逐级压缩 |
| **信息保留** | 🔴关键永久保留，🟡一般在月摘要时丢弃，🟢次要在周摘要后丢弃 |
| **原文保留** | 所有原始文件永久保留在磁盘，压缩只影响加载策略 |
| **溯源机制** | 长期记忆条目标注来源，可追溯到短期记忆原文 |
