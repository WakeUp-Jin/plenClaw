# 记忆模块更新工具的实现

---

## 一、工具定位

`edit_memory` 是记忆模块的核心写入工具，用于对长期记忆文件做**精确的局部修改**。

当前的 `memory` 工具只有两种操作：
- `append`：在某个 section 末尾追加一条，**只能新增，不能修改或删除**
- `rewrite`：整个文件重写，**破坏性强，容易丢信息**

`edit_memory` 填补中间的空白，支持**精准定位、局部替换**。

---

## 二、统一替换模型

所有的"增删改"操作，都映射为同一个替换操作：提供 `old_string`（要被替换的原文）和 `new_string`（替换后的内容）。

### 2.1 三种操作的映射方式

**新增（插入一行）**

通过包含锚点行来定位插入位置，让新行跟在锚点行后面：

```
old_string:
"- 修改代码前先说明\n"

new_string:
"- 修改代码前先说明\n- 不确定时要问我\n"
```

**删除（移除一行）**

把要删的行和上下文都包进 old_string，new_string 只保留上下文：

```
old_string:
"- 修改代码前先说明\n- 不确定时要问我\n- 每次只改一个文件\n"

new_string:
"- 修改代码前先说明\n- 每次只改一个文件\n"
```

**替换（修改内容）**

直接替换：

```
old_string:
"- 不确定时要问我\n"

new_string:
"- 不确定的时候必须问我，不要自己猜\n"
```

### 2.2 为什么要用统一替换而不是三个独立操作

独立的 insert/delete/replace 看起来更清晰，但有一个问题：insert 需要指定"在哪行之后插入"，而"哪行"本质上也是字符串匹配。统一替换把定位和修改合并在一个操作里，逻辑更简单，也和业界主流实现（Gemini CLI、Claude Code）一致。

## 三、工具参数设计

```python
edit_memory_def = {
    "name": "edit_memory",
    "description": """对长期记忆文件做精确的局部修改。

使用方式：
- 新增一行：old_string 写锚点行，new_string 写锚点行 + 新行
- 删除一行：old_string 写目标行及上下文，new_string 去掉目标行
- 修改内容：old_string 写旧内容，new_string 写新内容

重要约束：
1. 调用本工具前，必须先用 read_memory 读取文件的最新内容，
   再从读取结果中复制 old_string，不要凭记忆构造。
2. old_string 必须包含目标行前后至少 3 行上下文，确保在文件中唯一定位。
3. old_string 必须和文件内容完全一致，包括空格、缩进、换行符。
4. 如果工具报告找不到 old_string，不要修改 old_string 的核心内容，
   先重新读取文件确认当前内容，再重新构造。

                ],
                "description": "要修改的长期记忆文件名（不含路径和扩展名）",
            },
            "old_string": {
                "type": "string",
```

## 四、实现逻辑

### 4.1 执行流程

```
① 接收参数：file, old_string, new_string

② 读取文件最新内容
   → 文件不存在：报错返回（不支持创建，创建用 append）

③ 精确匹配：统计 old_string 在文件中出现的次数
   → 出现 0 次：报错（见风险一/二处理）
   → 出现 2+ 次：报错（见风险三处理）
   → 出现 1 次：进入步骤 ④

④ 执行替换：将文件内容中的 old_string 替换为 new_string

⑤ 写入文件，返回成功
```

### 4.2 精确匹配失败时的降级：宽松匹配

当精确匹配找不到时（步骤 ③ 出现 0 次），在报错前先尝试**宽松匹配**作为自动修复：

**宽松匹配规则**：对每一行，忽略行首行尾空白（trim），逐行比较是否一致。

```python
def _flexible_match_count(content: str, old_string: str) -> int:
    """逐行 trim 后比较，统计匹配次数。"""
    content_lines = content.split("\n")
    search_lines = [line.strip() for line in old_string.split("\n")]

    count = 0
    for i in range(len(content_lines) - len(search_lines) + 1):
        window = [line.strip() for line in content_lines[i:i + len(search_lines)]]
        if window == search_lines:
            count += 1
    return count
```

如果宽松匹配恰好命中 1 次，则自动用文件中实际的那段原文替换（而不是用传入的 old_string）。

**宽松匹配的作用**：缓解因行尾多余空格导致的精确匹配失败。只能处理空白差异，无法处理内容差异。

> ⚠️ 宽松匹配只是降级方案，不能完全解决空白字符差异问题（见第六节"已知问题"）。

## 五、风险处理

### 风险一：Stale Read（读了旧内容）

**描述**：Agent 基于上一轮读到的内容构造 old_string，但两轮之间文件已经被修改。

**症状**：精确匹配返回 0 次，宽松匹配也返回 0 次。

**处理方案**：报错，附带明确指令引导 Agent 重新读取。

```python
return json.dumps({
    "status": "error",
    "error_type": "not_found",
    "message": (
        "在文件中找不到 old_string 的匹配内容。"
        "文件可能已在本轮对话中被修改。"
        "请先调用 read_memory 读取文件的最新内容，"
        "再从读取结果中重新复制 old_string，然后重试。"
    ),
}, ensure_ascii=False)
```

**根本预防**：在工具描述里强制约束"每次调用前必须先 read_memory"。

---

### 风险二：空白字符差异

**描述**：行尾多余空格、Tab 和空格混用等导致精确匹配失败。

**处理方案**：降级到宽松匹配（见 4.2 节），能覆盖行首行尾空白差异。无法覆盖 Tab/空格混用等深层差异。

**预防方案（见第七节）**：在所有写入操作（包括 `append`）中统一格式规范，从源头减少差异积累。

**⚠️ 已知问题**：空白字符差异是**未完全解决的风险**，目前没有完美方案。宽松匹配只是部分缓解。

---

### 风险三：old_string 不唯一（多处匹配）

**描述**：old_string 过短，在文件中出现多次，无法确定要改哪一处。

**症状**：精确匹配返回次数 ≥ 2。

**处理方案**：报错，附带明确指令引导 Agent 加更多上下文。

```python
return json.dumps({
    "status": "error",
    "error_type": "ambiguous_match",
    "match_count": count,
    "message": (
        f"old_string 在文件中匹配到 {count} 处，无法唯一定位。"
        "请在 old_string 中加入更多前后行作为上下文，"
        "确保在文件中只有一处匹配，然后重试。"
        "前后各至少包含 3 行上下文是推荐做法。"
    ),
}, ensure_ascii=False)
```

**永远不要**在不唯一的情况下替换所有匹配，这会造成难以发现的数据污染。

---

### 风险四：Markdown 格式损坏

**描述**：替换操作导致 Markdown 结构损坏，比如标题前空行丢失、两个 section 挤在一起。

**处理方案**：**暂无好的自动修复方案。**

**当前缓解方式**：
- 在工具描述里约束 Agent 保持格式（new_string 末尾要有换行符）
- 写入规范（见第七节）统一格式，减少发生概率

**⚠️ 已知问题**：格式损坏是**未完全解决的风险**，依赖 Agent 的遵从度。

---

### 风险五：new_string 为空导致意外删除

**描述**：Agent 传入空的 new_string，导致 old_string 匹配到的内容被静默删除。

**处理方案**：**暂无防护。**

空 new_string 是合法的删除操作（用于删除某行）。禁止空 new_string 会损失正常删除能力，但允许又存在误用风险。

**当前缓解方式**：工具描述里明确说明"new_string 为空意味着删除，请谨慎"。

**⚠️ 已知问题**：意外删除是**未完全解决的风险**，可考虑未来引入独立的 `delete_memory` 工具，将删除和编辑显式分离。

## 六、错误返回格式汇总

所有报错都返回 JSON，带有 `error_type` 字段，便于 Agent 识别并采取对应行动：

| error_type | 含义 | Agent 应采取的行动 |
|---|---|---|
| `file_not_found` | 文件不存在 | 检查 file 参数是否正确 |
| `not_found` | old_string 未匹配 | 先 read_memory，重新复制 old_string |
| `ambiguous_match` | old_string 匹配多处 | 在 old_string 中加更多上下文 |
| `no_change` | old_string 和 new_string 一致 | 检查是否传错参数 |

成功时返回：

```json
{
  "status": "ok",
  "message": "已更新 user_instructions 中的 1 处内容"
}
```

## 七、写入规范

为了减少风险二（空白字符差异）的积累，**所有写入记忆文件的操作**（包括 `append`、`rewrite`、`edit_memory`）都应遵守以下格式规范：

1. **行尾不带空格**：每行写入前 strip 行尾空白
2. **统一使用 LF 换行符**：写入时将 `\r\n` 替换为 `\n`
3. **section 标题前后空行**：`##` 标题前有一个空行，标题后紧接内容（无额外空行）
4. **列表条目格式**：`- ` 开头（短横线 + 一个空格），条目末尾无空格
5. **文件末尾有且只有一个换行符**

这些规范在写入时由代码自动保证，不依赖 Agent 遵从。

---

## 八、与现有工具的关系

| 工具 | 适用场景 |
|------|---------|
| `memory append` | 向某个 section 新增一条，不需要定位 |
| `memory rewrite` | 整个文件结构需要大幅重组时（谨慎使用）|
| `edit_memory` | 修改或删除某条已有内容，需要精准定位 |

三个工具互补，`edit_memory` 不是用来替代 `append` 的。
