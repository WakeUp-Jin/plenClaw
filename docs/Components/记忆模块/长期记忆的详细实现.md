# 长期记忆实现方案文档

---

## 一、概述

### 1.1 定位

长期记忆是Agent对用户的持久认知，存储经过沉淀的、稳定的、重要的信息。它不随时间压缩或遗忘，只会通过定时更新机制进行增量维护。

### 1.2 核心机制

每天定时触发，并行启动多个专职更新Agent，每个Agent只负责一个长期记忆文件的更新判断。Agent通过唯一的文件写入工具完成更新，代码层保证操作的正确性。

### 1.3 设计原则

- **文件类型谨慎增加**：每种类型的长期记忆文件都有明确且不可替代的职责
- **并行更新**：每个文件的更新互不干扰，独立判断，并行执行
- **单一工具**：只提供一个文件写入工具，简化Agent的决策负担
- **代码保障**：文件操作的正确性由代码层保证，Agent只负责判断和生成内容

## 二、长期记忆文件体系

### 2.1 文件总览

| 文件 | 职责 | 加载策略 |
|------|------|----------|
| user_instructions.md | 用户对Agent的明确指令和规则 | 必加载 |
| user_profile.md | 用户基本信息与个人画像 | 必加载 |
| corrections.md | Agent犯过的错误记录 | 必加载 |
| project_context.md | 用户当前进行中的项目/事务 | 按需加载 |
| facts_and_decisions.md | 确认的事实与重要决策 | 按需加载 |
| topics_and_interests.md | 用户的兴趣与关注领域 | 按需加载 |
| interaction_patterns.md | Agent观察到的交互规律 | 按需加载 |

### 2.2 为什么是这7种类型

这7个文件覆盖了"了解一个人并为其服务"所需的全部认知维度：

```
① 他是谁？                → user_profile.md
② 他对我有什么要求？        → user_instructions.md
③ 我在哪里让他不满意过？    → corrections.md
④ 他在忙什么？             → project_context.md
⑤ 我们达成过哪些共识？      → facts_and_decisions.md
⑥ 他关心什么？             → topics_and_interests.md
⑦ 怎么和他沟通最顺畅？      → interaction_patterns.md
```

### 2.3 各文件职责边界

文件之间最容易混淆的边界需要明确：

**user_instructions vs interaction_patterns**

```
user_instructions = 用户明确说出来的要求
  "以后回复简短一点"
  "不要用any类型"

interaction_patterns = Agent观察推断出的规律
  用户每次收到长回复都只读前三行（用户从未明确说过）
  用户在下午的对话中更有耐心（Agent自己发现的）
```

**分界线：用户是否明确表达过。说出来的是instruction，观察到的是pattern。**

**corrections vs user_instructions**

```
corrections = 记录错误本身，提供"前车之鉴"视角
  "我建议用any，用户说不行"（记录的是：我犯了什么错）

user_instructions = 从纠错中提炼出的规则
  "不要使用any类型"（记录的是：以后怎么做）
```

**一次纠错可能同时触发两个文件的更新：corrections记录错误事件，instructions记录由此产生的规则。**

**project_context vs facts_and_decisions**

```
project_context = 项目的整体状态和进展
  "后台管理系统重构，已完成70%"

facts_and_decisions = 具体的决策点和确认的事实
  "2025-01-05 决定用zustand替代redux"
```

**project_context是面，facts_and_decisions是点。**

## 四、更新机制

### 4.1 触发方式

**每天定时执行，在当天最后一次对话结束后触发。**

```
判断条件：当天存在短期记忆日记录文件
触发动作：启动长期记忆更新流程
跳过条件：当天日记录为空或不存在（今天没有交互）
```

### 4.2 整体流程

```
定时触发
    │
    ▼
读取今日短期记忆日记录
    │
    ▼
日记录为空？ ── YES ──→ 跳过，结束
    │
    NO
    │
    ▼
读取所有长期记忆文件的当前内容
    │
    ▼
并行启动 N 个更新Agent（N = 长期记忆文件数量）
    │
    ├── Agent-1: instructions更新Agent
    ├── Agent-2: corrections更新Agent
    ├── Agent-3: profile更新Agent
    ├── Agent-4: project更新Agent
    ├── Agent-5: facts更新Agent
    ├── Agent-6: topics更新Agent
    └── Agent-7: patterns更新Agent
    │
    ▼
等待所有Agent完成
    │
    ▼
收集结果：
  - 哪些Agent调用了写入工具 → 执行文件更新
  - 哪些Agent返回"无需更新" → 跳过
    │
    ▼
生成更新日志
    │
    ▼
结束
```

### 4.3 单个更新Agent的输入输出

**输入**：

```
┌─────────────────────────────┐
│ 1. 系统提示词（该文件专属）    │
│ 2. 今日短期记忆日记录全文      │
│ 3. 该长期记忆文件的当前全文    │
└─────────────────────────────┘
```

**输出**：

```
┌──────────────────────────────────────┐
│ 选项A：调用 write_file 工具           │
│        输出更新后的完整文件内容         │
│                                      │
│ 选项B：不调用任何工具                  │
│        输出原因说明（代表无需更新）      │
└──────────────────────────────────────┘
```

### 4.4 唯一工具：write_file

**⚠️ 核心设计决策：只提供一个工具。**

```yaml
tool:
  name: write_file
  description: 将内容写入指定的长期记忆文件，完全覆盖原文件内容
  parameters:
    file_path:
      type: string
      description: 目标文件路径
    content:
      type: string
      description: 文件的完整新内容
```

**为什么只用一个覆盖写入工具，而不是 append / replace / remove？**

```
多工具方案的问题：
  - Agent需要判断"这次操作是追加还是替换还是删除"
  - 追加时要判断追加到哪个位置
  - 替换时要准确指定被替换的条目ID
  - 多次工具调用之间可能产生冲突
  - 工具越多，Agent出错概率越高

单工具方案的优势：
  - Agent的决策极其简单：要么输出新版文件，要么不更新
  - 不存在操作顺序问题
  - 不存在条目ID定位问题
  - 代码层只需要做一件事：把内容写入文件
  - Agent对最终文件有完整的掌控
```

**对"幻觉丢信息"的担忧如何解决？**

这是覆盖写入方案最大的风险：Agent输出新版文件时，可能遗漏原文件中的已有条目。

解决方式：

```
1. 在系统提示词中强调：
   "你必须在原文件基础上修改，不得遗漏任何已有条目。
    如果不需要更新，不要调用工具。"

2. 代码层做diff校验：
   - 写入前对比新旧文件
   - 如果发现已有条目被删除（非Agent主动标记删除），拒绝写入并告警
   - 只允许：新增条目、修改条目内容、Agent明确标记[移除]的条目消失

3. 备份机制：
   - 每次写入前自动备份旧文件
   - 保留最近N个版本（如5个版本）
```

### 4.5 代码层diff校验详细设计

这是保障数据安全的关键环节：

```python
def safe_write_long_term_memory(file_path, new_content):
    """安全写入长期记忆文件"""
  
    old_content = read_file(file_path)
  
    # 1. 提取新旧文件中的所有条目ID
    old_ids = extract_entry_ids(old_content)  # 如 {I001, I002, I003}
    new_ids = extract_entry_ids(new_content)  # 如 {I001, I002, I004}
  
    # 2. 检查是否有条目消失
    missing_ids = old_ids - new_ids  # {I003}
  
    if missing_ids:
        # 检查消失的条目是否被Agent标记为[移除]
        # 如果没有标记，说明是幻觉导致的遗漏，拒绝写入
        for mid in missing_ids:
            if not is_marked_for_removal(new_content, mid):
                log_warning(f"条目 {mid} 在新文件中消失且未标记移除，拒绝写入")
                return False
  
    # 3. 备份旧文件
    backup_file(file_path, old_content)
  
    # 4. 写入新文件
    write_file(file_path, new_content)
  
    # 5. 记录变更
    added = new_ids - old_ids
    log_change(file_path, added=added, removed=missing_ids)
  
    return True
```

## 五、更新Agent系统提示词

### 5.1 通用部分（所有更新Agent共享）

```markdown
# 通用规则

## 你的工作模式
你是一个长期记忆更新Agent。你会收到两份输入：
1. 今天的短期记忆日记录（用户今天的所有交互摘要）
2. 你负责的长期记忆文件的当前内容

你的任务是判断：今天的交互中，是否有信息需要更新到这个长期记忆文件。

## 决策规则
- 如果需要更新：调用 write_file 工具，输出更新后的完整文件内容
- 如果不需要更新：不调用任何工具，直接回复"无需更新"并说明原因

## 写入规则
- 你输出的是完整的文件内容，会覆盖原文件
- 你必须保留原文件中所有已有条目，除非你明确判断某条目需要移除
- 需要移除的条目，在最终输出中删除即可，代码层会做diff校验
- 新增条目需要分配新的编号，延续原有编号序列
- 保持原文件的格式风格不变
- 更新文件末尾的"最后更新"日期

## 判断原则
- 宁可漏掉，不可写错：不确定的信息不要写入
- 一次性的内容不写入：用户说"这次..."、"就这里..."的不算
- 已有的不重复写：原文件已有的信息不要重复添加
```

### 5.2 各Agent专属提示词

#### user_instructions 更新Agent

```markdown
# 用户指令集更新Agent

## 你负责的文件
user_instructions.md —— 记录用户对Agent的明确指令和规则。

## 什么信息应该写入这个文件？
用户明确表达的、希望Agent以后遵守的要求：
- "以后都要..."
- "每次帮我..."
- "记住不要..."
- "回复的时候请..."

- 一次性请求（"这次用英文注释"）
- 用户个人信息（属于 user_profile）
- 技术事实（属于 facts_and_decisions）
- Agent自己观察到的规律（属于 interaction_patterns）

## 冲突处理
如果新指令与已有指令矛盾：
- 用新指令替换旧指令
- 注意区分"推翻"和"补充"：
  - "注释改用英文" → 推翻 "注释用中文"，替换
  - "重要函数的注释要详细一些" → 补充，新增
```

## 你负责的文件
topics_and_interests.md —— 记录用户的兴趣和关注领域。

## 什么信息应该写入这个文件？
- 用户反复讨论的话题（频率高的）
- 用户明确表示感兴趣的领域
- 用户提到的爱好和生活兴趣

- 保持列表精简，同类话题合并

#### interaction_patterns 更新Agent

```markdown
# 交互模式更新Agent

## 你负责的文件
interaction_patterns.md —— 记录Agent观察到的交互规律和沟通偏好。

## 什么信息应该写入这个文件？
Agent从交互中观察到的规律：
- 用户偏好的回复长度、格式
- 用户在不同场景下的反应模式
- 有效的和无效的交互方式

## 重要区分
这个文件记录的是Agent的观察和推断，不是用户的明确要求。
用户明确说"以后回复短一点" → 属于 user_instructions
Agent观察到用户总是跳过长回复 → 属于 interaction_patterns

## 更新注意事项
- 需要多次观察才能确认一个模式，单次事件不写入
- 标注观察的置信度
- 如果用户行为模式发生变化，更新对应条目
```

### 6.3 日志用途

- **调试和审计**：出现问题时可以追溯什么时候改了什么
- **用户透明**：如果用户想知道"你记住了什么"，可以展示更新日志
- **回滚依据**：配合文件备份，可以精确回滚到某个版本

## 七、异常处理

### 7.1 Agent输出异常

```markdown
## 场景：Agent调用 write_file 但输出内容格式错误

处理方式：
1. 代码层做基本格式校验（是否有标题、是否有条目编号等）
2. 格式校验失败 → 拒绝写入，记录告警日志
3. 保持原文件不变
4. 在更新日志中标记为"更新失败：格式校验不通过"
```

### 7.2 diff校验不通过

```markdown
## 场景：Agent输出的新文件中，已有条目莫名消失

处理方式：
1. diff校验发现条目丢失
2. 拒绝写入，保持原文件
3. 记录告警：哪些条目消失了
4. 在更新日志中标记为"更新失败：疑似信息丢失"
5. 可选：触发一次重试（给Agent增加提示"请确保保留所有已有条目"）
```

### 7.3 并行冲突

```markdown
## 场景：两个Agent的更新结果存在逻辑冲突

实际上不会发生，因为每个Agent只负责一个文件，文件之间互不干扰。
这是并行方案的天然优势。

唯一可能的"逻辑关联"：一条纠错同时需要更新 corrections 和 instructions。
但这不是冲突，两个Agent各自独立写入各自的文件即可。
```

### 7.4 日记录过大

```markdown
## 场景：当天日记录非常长，超出更新Agent的上下文窗口

处理方式：
1. 如果日记录 > 更新Agent上下文窗口的50%，做预处理
2. 预处理：按时间段截取，分批给Agent处理
3. 或者：只提取带 🔴 重要性标注的条目给Agent
4. 确保更新Agent总是能完整接收输入
```

## 八、配置参数

```yaml
# 长期记忆更新配置

## 触发
trigger_mode: "daily"                    # 每天执行一次
trigger_timing: "after_last_session"     # 当天最后一次对话结束后

## 并行
parallel_agents: true                    # 并行执行更新Agent
max_parallel: 7                          # 最大并行数

## 文件控制
max_file_size_tokens: 3000               # 单个长期记忆文件最大Token数
max_file_count: 7                        # 长期记忆文件最大数量

## 安全
enable_diff_check: true                  # 启用diff校验
backup_versions: 5                       # 保留最近5个备份版本
retry_on_failure: true                   # 校验失败时重试
max_retry: 1                             # 最大重试次数

## 日志
enable_update_log: true                  # 启用更新日志
log_retention_days: 30                   # 日志保留30天
```
