## 二、逐点补充与问题说明

### 2.1 关于Skill形式存储（你的第1点）

**优点：**
- 用户可以跨项目、跨设备迁移记忆
- 可以分享给他人（比如团队共享某些记忆模板）
- 符合你已有的Skill渐进式加载架构

**需要补充的：**
- 记忆Skill应该标记为**特殊类型**，和普通功能Skill区分开
- 建议在SKILL.md的元信息中加一个字段，比如 `type: memory`
- **隐私问题**：记忆可能包含敏感信息，移植时需要考虑是否需要脱敏选项

**潜在缺点：**
- Skill的粒度设计原本是「功能模块」，记忆是「持续增长的数据」，两者的生命周期不同。功能Skill相对稳定，记忆Skill需要频繁写入，这可能对你的Skill管理机制产生压力
- 建议：记忆Skill内部的文件读写需要一套独立的规则，不完全等同于普通Skill

### 2.2 关于短期记忆与长期记忆（你的第2点）

**短期记忆我的理解：**
你的助手Agent没有传统的「会话」边界，所以你用短期记忆来替代会话历史。本质上它是一个**滚动的交互日志**。

**长期记忆需要细化分类，我建议分为以下几种类型：**

```
长期记忆/
├── user_profile.md          # 用户画像（偏好、习惯、身份信息）
├── user_instructions.md     # 用户指令（类似CLAUDE.md）
├── project_context.md       # 项目/工作上下文
├── learned_patterns.md      # 学到的交互模式和经验
├── facts_and_decisions.md   # 用户明确告知的事实和决策
└── corrections.md           # 用户纠正过的错误，避免重犯
```

**为什么要这样分？**

参考Claude Code的Memory设计，它存储的主要是：
- 用户偏好和风格要求
- 项目技术细节
- 纠正后的行为模式

但你的助手Agent范围更广（不限于编程），所以需要更多类型。每种类型的**更新频率和重要性不同**，分开存储方便后续的加载决策和清理策略。

**用户指令（user_instructions.md）需要特别说明：**
- 这个文件的地位最高，类似宪法
- 它不是Agent自动整理的，而是**用户主动写入或确认后写入**的
- 每次都应该加载（属于你说的第二层渐进式加载的一部分）

**潜在缺点：**
- 长期记忆分类太多会增加整理Agent的判断负担
- 建议初期可以先用少量分类（比如3个），后续根据实际使用情况扩展

### 2.3 关于溯源机制（你的第3点）

**这个设计非常好，补充几点：**

```markdown
<!-- 长期记忆文件中的溯源格式示例 -->
## 用户偏好：回复风格

用户偏好简洁直接的回复风格，不喜欢过多的客套话。

> 来源：short_term/2025-01-15.md#L42-L58
> 提取时间：2025-01-16T03:00:00
> 置信度：高（用户明确要求）
```

**补充建议：**
- 加一个**置信度**字段：用户明确说的 vs Agent自己推断的，权重不同
- 加一个**提取时间**：方便后续判断信息是否过时
- 溯源路径应该是精确到行号或段落锚点，而不仅是文件名

**潜在缺点：**
- 短期记忆文件如果被压缩或清理了，溯源链就断了
- **解决方案**：短期记忆压缩时保留原文件的归档副本（只是不再主动加载），或者在长期记忆的溯源信息中同时保存关键原文片段

### 2.4 关于短期记忆的分片与加载策略（你的第4点）

**按日期分片的优点：**
- 天然的时间索引
- 方便压缩和清理
- 加载时可以灵活选择范围

**关于加载策略，这是一个关键决策：**

| 策略 | 优点 | 缺点 |
|------|------|------|
| 加载全部 | 信息不会遗漏 | Token消耗大，30天后不可持续 |
| 加载最近7天 | 平衡性好 | 可能丢失上周重要上下文 |
| 加载最近30天 | 信息量充足 | Token消耗较大 |
| 智能加载（推荐） | 最优解 | 实现复杂度高 |

**我的建议——分层加载策略：**

```
第一优先级：今天的短期记忆（完整加载）
第二优先级：昨天的短期记忆（完整加载）
第三优先级：最近7天的短期记忆（加载摘要/压缩版）
第四优先级：7-30天的短期记忆（仅在需要时通过溯源触发加载）
第五优先级：30天以上（归档，正常流程不加载，只通过长期记忆溯源访问）
```

**关于压缩功能的补充：**
- 压缩不应该是删除，而是生成摘要版本，原文归档
- 压缩的触发时机：建议每天凌晨（或用户指定时间）由整理Agent执行
- 压缩后的文件命名建议：`2025-01-15.md`（原文）→ `2025-01-15.summary.md`（摘要）

**潜在缺点：**
- 如果用户一天内交互非常密集，单日文件可能很大
- 建议：单日文件超过一定Token数时，自动拆分为上午/下午，或按交互话题分段

### 2.6 关于四层渐进式加载（你的第6点）

**你的四层设计整理：**

```
第一层：SKILL.md 元信息（Skill列表扫描时读取，判断是否需要这个记忆Skill）
第二层：SKILL.md 正文 + 长期记忆文件索引（每次必定加载）
第三层：具体的长期记忆文件（按需加载）
第四层：长期记忆溯源的短期记忆原文（深度回溯时加载）
```

**补充建议：**

第二层中的「长期记忆文件索引」应该包含足够的信息让Agent决定是否需要深入第三层：

```markdown
## 长期记忆文件索引

### user_profile.md
- 描述：用户的身份信息、偏好、习惯
- 最后更新：2025-01-16
- 加载时机：每次会话开始时加载
- 大小：约 500 tokens

### corrections.md
- 描述：用户纠正过的AI行为记录
- 最后更新：2025-01-15
- 加载时机：当AI即将执行与历史纠正相关的操作时
- 大小：约 300 tokens

### project_context.md
- 描述：当前项目的技术细节和进度
- 最后更新：2025-01-16
- 加载时机：用户讨论项目相关话题时
- 大小：约 800 tokens
```

**潜在缺点：**
- 四层加载意味着最坏情况下需要4次文件读取操作，延迟可能影响用户体验
- Agent需要足够聪明才能判断「什么时候该深入下一层」
- **缓解方案**：某些高优先级的长期记忆文件（如user_profile、user_instructions）可以标记为「始终加载」，跳过判断直接进入第二层

## 三、我发现的你的设计中缺少的部分

### 3.1 记忆的遗忘/衰减机制

你设计了记忆的写入和压缩，但没有提到**遗忘**。

- 短期记忆需要过期清理策略（比如90天后自动归档到冷存储）
- 长期记忆中的某些信息可能过时（用户换了工作、项目结束了）
- **建议**：长期记忆的每条信息加一个`last_referenced`时间戳，超过一定时间未被引用的信息，整理Agent可以建议降级或移除

### 3.2 记忆冲突的处理

用户可能在不同时间说了矛盾的话：
- 1月1日："我喜欢详细的解释"
- 1月15日："你能不能简洁点"

**需要一个冲突解决策略**：
- 时间优先：新的覆盖旧的
- 明确优先：用户明确指示 > Agent推断
- 存疑标记：整理Agent不确定时标记为冲突，等待用户确认

### 3.3 用户对记忆的控制界面

虽然你说了用户有最大权限，但具体的控制方式需要考虑：
- 用户能查看所有存储的记忆吗？
- 用户能删除特定记忆条目吗？
- 用户能手动添加/修改长期记忆吗？
- 用户能暂停记忆功能吗？

### 3.4 记忆写入的触发时机

什么时候将交互内容写入短期记忆？
- 每次对话结束后？
- 实时写入？
- 这个需要明确，因为你的助手Agent没有会话的概念

**我的建议**：
- 短期记忆实时追加写入（每次交互完成后立即写入当天的文件）
- 长期记忆定期整理（可以设定触发条件：每天一次，或短期记忆积累到一定量时触发）

### 3.5 记忆的作用域

如果用户有多个项目/领域的Skill，记忆是全局的还是项目级的？

**建议**：
```
memory/                      # 全局记忆Skill
├── SKILL.md
├── long_term/
│   ├── user_profile.md     # 全局
│   ├── user_instructions.md # 全局
│   └── ...
├── short_term/
│   └── ...
└── project_memories/        # 项目级记忆（可选）
    ├── project_a/
    └── project_b/
```

## 四、长期记忆整理Agent的系统提示词

参考Claude Code的Memory实现，结合你的架构设计：

```markdown
# 长期记忆整理Agent - 系统提示词

## 你的角色

你是一个记忆整理专家。你的职责是阅读用户的短期记忆（近期交互日志），
从中提取有价值的信息，整理到长期记忆文件中。

你不直接与用户对话。你在后台工作，默默维护用户的记忆库。

## 你的工作原则

### 提取原则
1. **只提取有长期价值的信息**：不是所有对话都值得记住。忽略一次性的问答、闲聊、
   已经完全解决且不会复现的问题
2. **用户明确表达 > 你的推断**：如果用户说"记住这个"或"以后都这样做"，
   这是最高优先级。如果你从对话模式中推断出偏好，标记为推断，置信度降低
3. **保持简洁**：长期记忆应该是精炼的要点，不是对话的复制粘贴。
   但要保留足够的上下文让未来的AI能理解这条记忆的含义
4. **不要过度推断**：如果用户只是某一次说了某个偏好，不要急于写入长期记忆。
   等出现2-3次一致的模式再提取
5. **尊重隐私**：敏感的个人信息（密码、证件号等）绝不写入记忆

### 分类原则
将提取的信息分类存入对应的长期记忆文件：

- **user_profile.md**：用户是谁、做什么工作、有什么特点
  - 写入时机：用户透露个人信息、偏好、身份相关内容时
  
- **user_instructions.md**：用户对AI行为的明确指令
  - 写入时机：用户说"以后..."、"每次都..."、"不要..."等指令性表达时
  - ⚠️ 注意：这个文件的修改需要格外谨慎，建议标记为"待用户确认"
  
- **project_context.md**：项目相关的技术细节、进度、架构决策
  - 写入时机：用户讨论项目结构、技术选型、遇到的问题和解决方案时
  
- **learned_patterns.md**：从交互中学到的模式
  - 写入时机：发现用户反复出现的行为模式、提问方式、工作习惯时
  - 需要至少2次一致的观察才能写入
  
- **facts_and_decisions.md**：用户明确告知的事实和做出的决策
  - 写入时机：用户陈述事实、做出决定、确认方案时
  
- **corrections.md**：用户纠正AI行为的记录
  - 写入时机：用户说"不对"、"不是这样"、"我要的是..."时
  - 这个文件极其重要，直接影响AI未来的行为

### 溯源原则
每条记忆必须标注来源：
```
> 来源：short_term/2025-01-15.md#相关段落描述
> 提取时间：{当前时间}
> 置信度：高/中/低
> 类型：用户明确表达 / Agent推断 / 用户纠正
```

### 冲突处理原则
1. 发现新信息与已有记忆矛盾时：
   - 如果新信息来自用户明确表达：更新旧记忆，保留旧记忆的痕迹（用删除线或注释）
   - 如果新信息来自推断：标记冲突，不覆盖旧记忆，两条都保留并标记待确认
2. 在更新记录中注明变更历史

### 清理原则
1. 检查长期记忆中`last_referenced`超过60天的条目，标记为"可能过时"
2. 合并重复或高度相似的记忆条目
3. 移除已经不再适用的信息（如已结束的项目上下文）

## 你的工作流程

1. 读取最近尚未处理的短期记忆文件
2. 读取当前所有长期记忆文件
3. 逐条分析短期记忆中的对话
4. 判断哪些信息有长期价值
5. 将有价值的信息分类写入对应的长期记忆文件
6. 处理可能的冲突和重复
7. 更新 SKILL.md 中的文件索引信息（最后更新时间、大小等）
8. 标记已处理的短期记忆文件（避免重复处理）

## 你不应该做的事

- 不要编造用户没有说过的信息
- 不要存储一次性的、没有复用价值的信息
- 不要存储敏感信息（密码、密钥、证件号）
- 不要在没有充分证据的情况下覆盖已有的长期记忆
- 不要改变用户指令文件（user_instructions.md）中用户亲自写入的内容，
  只能在文件末尾的"Agent建议区"添加建议
```

```markdown
# 用户记忆模块

本Skill管理用户与AI助手之间的交互记忆，分为短期记忆和长期记忆两部分。

## 加载策略

- 本文件（SKILL.md）：每次交互必定加载
- user_instructions.md：每次交互必定加载
```

### 我的解决方案：以容量为主，时间为辅

```
│   └── 短期记忆预算：~25K tokens ★★
├── 当前对话区：~70K tokens（留给本次交互的来回对话）
└── 缓冲区：~13K tokens（安全余量）
```

**短期记忆的25K Token预算，动态填充**：

```
步骤1：加载今天的完整记录 → 消耗 X tokens
步骤2：剩余预算 = 25K - X
步骤3：加载昨天的完整记录 → 消耗 Y tokens
步骤4：剩余预算 = 25K - X - Y
步骤5：继续往前加载，直到预算用完
步骤6：预算用完时，剩下的天数只加载摘要版
步骤7：如果连摘要都放不下了，就停止
```

3. **压缩应该按周为单位**，而不是简单的百分比裁剪
4. **摘要文件替代原文件加载**，有摘要就加载摘要，没有就加载原文
5. **多轮压缩怎么叠加**，这是你最想讨论的核心问题

## 你的第1点引发的重要思考

你说得非常对。让我重新理解一下这个模型：

```
传统Agent的做法：
  系统提示词 + 历史消息 + 当前对话 = 上下文
  需要为"当前对话"预留空间

你的助手Agent：
  系统提示词 + Skills + 短期记忆 + 当前交互 = 上下文
  但"当前交互"结束后会写入短期记忆
  所以短期记忆和当前交互本质上是同一条河流
```

这意味着：**短期记忆的Token预算，其实就是"Agent能记住多少近期内容"的总容量**。当前对话也好、昨天的记录也好，都在争夺这个容量。

所以更准确的模型应该是：

```
模型总上下文窗口（如128K）
├── 固定开销：系统提示词 + Skills + 长期记忆（相对固定，如15K）
└── 短期记忆总容量：剩余的全部空间（如113K）
    这113K里面装的是：
    ├── 已加载的历史短期记忆文件/摘要
    └── 当前正在进行的对话（每一轮对话都在消耗这个空间）
```

**当前对话不断增长 → 挤压历史短期记忆的空间 → 触发压缩**

这个理解对吗？如果对的话，压缩的触发条件就很自然了：

> **当 固定开销 + 已加载短期记忆 + 当前对话 逼近上下文窗口上限时，触发压缩。**

## 加载策略（下一次新交互开始时）

上面讨论的是**对话过程中的压缩**。还有一个场景：**新的一天开始（或新的交互开始），应该怎么加载？**

```markdown
## 新交互的短期记忆加载策略

### 原则
从最近的时间往回加载，直到预算用完。
遇到摘要文件就加载摘要，不加载对应的原文。

### 具体步骤

1. 计算短期记忆总预算 = 上下文窗口 - 固定开销
2. 设置加载预算 = 总预算 × 60%（为今天的对话预留40%增长空间）★ 见下面说明
3. 扫描 short_term/ 目录，按时间倒序排列所有文件
4. 逐个加载：

   for each 时间点（从今天往回）:
       if 存在该时间点的摘要文件:
           加载摘要文件
       elif 存在该时间点的日记录:
           加载日记录
       
       if 已用预算 ≥ 加载预算:
           停止加载
           break
```

### 关于"预留40%"的说明

你说得对，不应该为未来对话预留**固定**空间。但新交互开始时，有一个特殊情况：

**这个40%不是"预留"，而是"弹性区间"**：

```
初始加载时：用60%预算加载历史短期记忆
对话过程中：当前对话不断增长，占用剩余的40%
当对话增长到85%触发线时：压缩最旧的内容，释放空间
对话继续增长 → 再压缩 → 再释放 → 循环
```

所以这40%不是浪费，它会被当前对话逐渐填满，然后通过压缩来动态平衡。

如果你觉得60%太保守，可以调高到70%甚至80%，这只影响**第一次压缩被触发的时机早晚**。

```
  从今天往回，逐天加载日记录
  遇到周摘要覆盖的日期范围 → 加载周摘要，跳过对应日记录

对于更早的月份（2024-12）：
  先看有没有 month_xxxx-xx.summary.md
  有 → 直接加载月摘要，整个文件夹不再深入
  没有 → 进入文件夹，按周摘要/日记录的逻辑加载

直到预算用完停止
```

**优点：**
- 目录结构清晰，不会有100+文件平铺的问题
- 不是archive的概念，日记录和摘要住在一起，就是按月归类
- Agent扫描时可以以月为单位快速跳过（有月摘要就跳过整个月）
- 溯源时进入对应月份文件夹就能找到原文

## 我的建议

如果你坚持不要子文件夹，方案A可以工作，但长期使用后目录会臃肿。

如果你能接受**按月分文件夹（不是archive，只是整理）**，方案B在加载效率和可维护性上都好很多。
