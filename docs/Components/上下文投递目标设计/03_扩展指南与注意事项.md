# 扩展指南与注意事项

## 如何新增一个上下文模块

比如要添加一个 RAG（检索增强）模块，将检索到的文档片段注入上下文。

### Step 1: 创建模块

```python
# src/core/context/modules/rag_context.py
from core.context.base import BaseContext
from core.context.types import ContextParts, SystemPart

class RAGContext(BaseContext[str]):
    """检索增强上下文：将检索到的文档片段注入 system message。"""

    def format(self) -> ContextParts:
        docs = self.get_all()
        if not docs:
            return ContextParts()

        text = "\n---\n".join(docs)
        return ContextParts(system_parts=[
            SystemPart(
                tag="retrieved_documents",
                description="以下是根据用户问题检索到的相关文档片段，请基于这些信息回答",
                content=text,
            ),
        ])
```

关键决策：内容投递到 `system_parts` 还是 `message_items`？

| 投到 system_parts | 投到 message_items |
|-------------------|-------------------|
| 静态/背景性质的内容 | 对话流中的消息 |
| 不需要对话轮次顺序 | 需要保留时间顺序 |
| 如：提示词、记忆、摘要、检索文档 | 如：用户消息、助手回复、工具调用 |

### Step 2: 注册到 ContextManager

在 `ContextManager.__init__()` 中接收新模块，在 `_collect_parts()` 的循环中按适当顺序加入。

### Step 3: 确定顺序

system message 内部各段的顺序会影响 LLM 的理解。建议顺序：

1. **核心指令**（system_prompt）— 告诉 LLM 它是谁、怎么做
2. **RAG 文档**（retrieved_documents）— 当前回答的参考资料
3. **长期记忆**（long_term_memory）— 用户画像和偏好
4. **对话摘要**（conversation_summary）— 历史背景

## 注意事项

### 1. 空内容过滤

`get_context()` 在合并 system_parts 时会过滤掉 `content.strip()` 为空的部分：

```python
rendered = SYSTEM_PART_SEPARATOR.join(
    part.render() for part in system_parts if part.content.strip()
)
```

所以模块的 `format()` 可以放心返回空内容的 SystemPart，不会产生空 XML 标签。

### 2. 不要在 format() 中直接构造 message dict

```python
# ❌ 错误
def format(self):
    return [{"role": "system", "content": self.text}]

# ✅ 正确
def format(self) -> ContextParts:
    return ContextParts(system_parts=[
        SystemPart(tag="my_module", description="...", content=self.text)
    ])
```

`format()` 返回的是 `ContextParts`，不是 message dict。message dict 只在 `get_context()` 的最终步骤产生。

### 3. description 的使用原则

- 核心提示词（system_prompt）：留空。它本身就是指令，不需要"解释指令"
- 其他模块：写一句简明的行为指引，告诉 LLM 如何使用这段内容
- 示例：
  - `"以下是你对用户的长期记忆，请基于这些信息个性化回复"`
  - `"以下是之前对话的压缩摘要"`
  - `"以下是根据用户问题检索到的相关文档片段，请基于这些信息回答"`

### 4. XML 标签命名规范

- 使用 `snake_case`
- 含义清晰，一看就知道是什么模块的内容
- 和模块名保持一致关系：`LongTermMemoryContext` → `long_term_memory`

### 5. 压缩发生在 ContextItem 层面

压缩操作（裁剪旧消息、LLM 摘要）作用于 `ShortTermMemoryContext` 内部的 `ContextItem` 列表。压缩产生的摘要文本存储在 `_summary` 字段中，在 `format()` 时才被包装为 `SystemPart` 投递到 system。

```
压缩前: _items = [item1, item2, ..., item20]
压缩后: _summary = "摘要文本...",  _items = [item18, item19, item20]

format() 时:
  _summary → SystemPart(tag="conversation_summary") → system_parts
  _items   → message_items
```

## 改动的文件清单

| 文件 | 改动 |
|------|------|
| `src/core/context/types.py` | 新增 `SystemPart` 和 `ContextParts` 数据类 |
| `src/core/context/base.py` | `format()` 返回类型改为 `ContextParts` |
| `src/core/context/modules/system_prompt.py` | `format()` 返回 `ContextParts`，内容放入 `system_parts` |
| `src/core/context/modules/long_term_memory.py` | `format()` 返回 `ContextParts`，删除 `MEMORY_PREAMBLE`，描述移入 `SystemPart.description` |
| `src/core/context/modules/short_term_memory.py` | `format()` 返回 `ContextParts`，摘要放 `system_parts`，消息放 `message_items` |
| `src/core/context/manager.py` | 新增 `_collect_parts()`，重写 `get_context()` 和 `_build_context_items()`，删除 `_to_messages()` |
