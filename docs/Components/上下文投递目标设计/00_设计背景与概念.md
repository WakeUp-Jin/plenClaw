# 设计背景与概念

## 问题

之前的上下文模块（`SystemPromptContext`、`LongTermMemoryContext`、`ShortTermMemoryContext`）的 `format()` 方法都直接返回 `list[ContextItem]`，每个模块自己设置 `role="system"`。这导致最终发送给 LLM 的消息序列中出现 **2~3 条 system 消息**。

问题的本质有两层：

### 1. 内部数据结构和 LLM 消息格式耦合

各模块通过 `ContextItem(role="system")` 直接决定了 LLM message 的 role。但 `role` 是 LLM API 的概念，不应该是内部数据结构的概念。模块应该只关心"我的内容放在哪里"，而不是直接操控 API 格式。

### 2. 多条 system 消息的兼容性问题

- 部分模型（尤其是国产模型）对多个 system 消息支持不稳定，可能报错或忽略后续 system 消息
- 多个 system 消息在语义上是割裂的，模型无法将它们视为统一的指令
- prefix caching 要求 system 消息内容稳定且连续，多条会降低缓存命中率

### 改动前的数据流

```
SystemPromptContext.format()   → [ContextItem(role="system")]  ─┐
LongTermMemoryContext.format() → [ContextItem(role="system")]  ─┤→ items → to_message() → LLM
ShortTermMemoryContext.format()→ [ContextItem(role="system"),  ─┘
                                  ContextItem(role="user"), ...]
```

最终 message 数组：

```
[system] 核心提示词...
[system] 长期记忆...         ← 第 2 条 system
[system] 压缩摘要...         ← 第 3 条 system（有压缩时）
[user] ...
[assistant] ...
```

## 解决方案：投递目标设计

### 核心思路

**概念上可以有任意多种上下文模块，但最终拼接到 message 数组的投递目标只有两种：**

| 投递目标 | 含义 | 最终效果 |
|---------|------|---------|
| **system** | 内容合并进唯一的 system message | 多个模块的 system 内容用 XML 标签分段，合并为一条 |
| **messages** | 内容作为独立 message 放入消息列表 | user/assistant/tool 消息按顺序排入数组 |

### 各模块的投递策略

| 模块 | 投递到 system | 投递到 messages |
|------|-------------|----------------|
| SystemPromptContext | 核心提示词 + 动态 segment | — |
| LongTermMemoryContext | 用户记忆 | — |
| ShortTermMemoryContext | 压缩摘要 | 对话消息（user/assistant/tool） |

### 改动后的数据流

```
SystemPromptContext.format()   → ContextParts(system_parts=[SP1])         ─┐
LongTermMemoryContext.format() → ContextParts(system_parts=[SP2])         ─┤→ _collect_parts()
ShortTermMemoryContext.format()→ ContextParts(system_parts=[SP3],         ─┘
                                               message_items=[item, ...])
    │
    ▼  get_context()
system_parts → SP1.render() + SP2.render() + SP3.render() → 1 条 system msg
message_items → 逐个 to_message() → N 条对话 msg
    │
    ▼
[{"role": "system", "content": "<system_prompt>...\n\n<long_term_memory>...\n\n<conversation_summary>..."},
 {"role": "user", ...},
 {"role": "assistant", ...},
 ...]
```

## 关键概念

### 内部状态 vs LLM 消息格式

- **ContextItem**：内部流转数据结构，携带 source、priority、thinking、usage 等元数据
- **Message dict**：LLM API 格式（role/content/tool_calls），只在最终组装时生成
- **SystemPart**：system 级内容的内部表示，带 XML 标签和描述

模块内部操作 ContextItem 和 SystemPart，**不直接操作 message dict**。

### 为什么用 XML 标签标注

合并后的 system message 如果没有标注，各段内容混在一起，模型不知道哪段是什么。通过 XML 标签：

1. LLM 能清晰识别每段内容的边界和含义
2. `description` 属性提供行为指引（如"请基于这些信息个性化回复"）
3. 调试时一眼就能看出哪段内容来自哪个模块
